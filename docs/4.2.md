Based on section 4.2 of the specs — **Sales Pipeline & Deal Management** — here are the user stories and acceptance criteria for each listed feature.

---

## Feature 1: Multiple configurable pipelines

**Story:** As a sales manager, I can create and configure multiple pipelines so that different sales motions (new business, renewals, upsells) have their own stages and rules.

Acceptance criteria:
- An admin or manager can create a named pipeline with a description and an ordered list of stages.
- Each stage has a name, a probability percentage (0–100), and an optional description.
- Stage order is configurable via drag-and-drop in the pipeline settings screen. Reordering does not affect existing deals already in those stages.
- A pipeline can be marked as the default. New deals default to this pipeline unless the agent selects another.
- A pipeline can be archived when no longer in use. Archived pipelines are hidden from deal creation but remain accessible for reporting on historical deals.
- Deleting a pipeline is only permitted if it contains zero deals. Otherwise the system returns a validation error directing the manager to archive instead.
- At least one pipeline must exist at all times. The last pipeline cannot be deleted or archived.
- Pipeline configuration is restricted to manager role and above.

## Feature 2: Drag-and-drop Kanban board

**Story:** As an agent, I can view my pipeline as a Kanban board and move deals between stages by dragging so that I can manage my pipeline visually in real time.

Acceptance criteria:
- The board renders one column per stage in the selected pipeline, ordered by stage sequence.
- Each deal card shows the deal title, account name, value, expected close date, and owner avatar.
- A deal is moved by dragging its card to another column. On drop, the deal's stage updates immediately in the UI optimistically, then confirms via an API call.
- If the API call fails, the card snaps back to its original column and an error toast is shown.
- When another agent moves a deal while the current user has the board open, the board updates in real time via WebSocket without requiring a page refresh.
- Columns show the total deal count and total weighted value (deal value × stage probability) in the column header.
- The board can be filtered by owner, close date range, and deal value range without leaving the board view.
- Deals can also be opened from the board into a detail slide-over panel without navigating away from the board.

## Feature 3: Deal stage automations

**Story:** As a sales manager, I can configure automations that trigger when a deal enters a stage so that follow-up actions happen consistently without manual effort.

Acceptance criteria:
- Automations are configured per pipeline stage. Supported trigger: deal enters stage.
- Supported actions are: create an activity (task or call) assigned to the deal owner or a specified user, send an email notification to the deal owner or a specified address, and fire an outbound webhook to a configured URL.
- Each automation action has a configurable delay — immediately, after 1 hour, after 1 day, or after N business days.
- Multiple actions can be configured for the same stage trigger. They execute in the defined order respecting their individual delays.
- Automation jobs are queued. A failed job retries up to three times with exponential backoff before being marked failed and alerting the admin.
- A deal can be excluded from stage automations by toggling an override flag on the deal. This is logged in the audit trail.
- Automation configuration is restricted to manager role and above. Agents can view configured automations but cannot modify them.

## Feature 4: Probability weighting and revenue forecasting

**Story:** As a sales manager, I can view a weighted revenue forecast so that I can predict expected revenue from the current pipeline with realistic probability adjustments.

Acceptance criteria:
- Each deal's weighted value is calculated as deal value multiplied by its current stage probability percentage.
- The forecast view shows total pipeline value (unweighted), total weighted value, and a breakdown by pipeline stage.
- The forecast can be filtered by owner, team, close date range, and pipeline.
- A time-bucketed view groups deals by expected close month, showing weighted and unweighted totals per month.
- Historical win rate per stage is calculated from closed deals in the last 12 months and shown alongside the configured probability for comparison.
- The forecast page refreshes its data on load and can be manually refreshed. It does not update in real time.
- Forecast data is available via the analytics API endpoint for export to external BI tools.
- Forecast is visible to agents (own deals only), managers (team deals), and admins (all deals).

## Feature 5: Win/loss reasons capture

**Story:** As a sales manager, I can require agents to record a reason when closing a deal as won or lost so that I can analyse patterns and improve the sales process.

Acceptance criteria:
- When a deal is moved to the Closed Won or Closed Lost stage, a modal appears requiring the agent to select a reason from a predefined list and optionally enter a free-text note.
- The reason list is configurable by admins. Won reasons and lost reasons are maintained as separate lists. Each list must contain at least one active reason.
- Closing a deal without selecting a reason is blocked — the stage move does not complete until the modal is submitted.
- The selected reason and note are stored on the deal record and visible in the deal detail view.
- Reason codes appear in the analytics pipeline report, showing frequency and total value by reason for won and lost deals over a configurable date range.
- Reason lists can have entries added or deactivated by admins. Deactivated reasons remain visible on historical deals but do not appear in the selection modal for new closures.

## Feature 6: Activity logging on deals

**Story:** As an agent, I can log activities (calls, emails, meetings, tasks) directly against a deal so that all deal-related interactions are captured in one place.

Acceptance criteria:
- From a deal detail view, an agent can create an activity of type call, email, meeting, or task.
- Each activity requires a subject, type, and due date. Outcome, duration, and notes are optional.
- A completed activity requires a completion date and an outcome selection (positive, neutral, negative, follow-up required).
- Activities logged on a deal appear in both the deal's activity list and the associated contact's timeline.
- Overdue activities (past due date and not completed) are highlighted visually on the deal card and in the agent's activity list.
- An agent receives an in-app notification 30 minutes before a scheduled activity is due, if the activity has a specific time set.
- Activities can be edited or deleted by the creating agent or a manager. Deletion writes an audit log entry.
- The deal detail view shows a summary count of open, overdue, and completed activities.

## Feature 7: Quote and proposal generation

**Story:** As an agent, I can generate a formatted PDF quote or proposal from a deal so that I can send professional pricing documents to prospects without leaving the CRM.

Acceptance criteria:
- From a deal detail view, an agent can initiate quote generation by selecting a quote template.
- Quote templates are configured by admins and support variable placeholders for deal value, product line items, contact name, account name, validity date, and agent signature block.
- Product line items are pulled from the deal's associated products. The agent can adjust quantities and unit prices before generating.
- The generated PDF is stored in S3 and linked to the deal record. Multiple quote versions can exist on a deal; each is versioned and timestamped.
- The agent can download the PDF, copy a shareable link, or send it directly via email from the CRM. Sending via email logs an outbound interaction against the deal.
- Quote generation is logged in the audit trail with the template used, the agent, and the timestamp.
- A quote can be marked as accepted or declined by the agent when feedback is received. Marking a quote as accepted optionally triggers the deal to advance to the next stage.

## Feature 8: Deal collaboration — comments and mentions

**Story:** As an agent, I can leave comments on a deal and mention teammates so that deal context is shared and the right people are looped in without email.

Acceptance criteria:
- A comment thread is available on every deal detail view. Any user with deal access can post a comment.
- Comments support plain text and @mention syntax. Typing @ presents a dropdown of team members filtered by the typed name.
- A mentioned user receives an in-app notification and an email notification linking directly to the deal and the comment. Email notification can be disabled per user in notification preferences.
- Comments are ordered chronologically, newest at the bottom.
- A comment can be edited by its author within 15 minutes of posting. After 15 minutes it is locked. Edited comments show an "edited" indicator.
- A comment can be deleted by its author or a manager. Deleted comments are replaced with a "comment deleted" placeholder showing the deletion timestamp and actor.
- Comment activity does not appear in the contact timeline — it is internal deal collaboration only.
- The deal detail view shows a badge count of unread comments for the current user since their last visit.

---

## Feature 9: Demo and trial management

**Story:** As an agent, I can schedule and track product demos and trial periods linked to a deal so that prospect evaluation activities are managed within the pipeline.

Acceptance criteria:
- A demo or trial record can be created from a deal. It requires a type (demo or trial), a scheduled date, and an assigned agent.
- A trial record additionally requires a start date, end date, and trial scope notes.
- Scheduled demos appear in the shared team calendar and in the assigned agent's activity list.
- When a demo or trial start date arrives, the assigned agent receives an in-app and email reminder.
- Three days before a trial end date, the deal owner receives an automated reminder to follow up with the prospect.
- Demo and trial records have a status: scheduled, completed, no-show, or converted. The agent updates the status after the event.
- Completing a demo or trial with status converted optionally triggers the deal to advance to the next configured pipeline stage.
- Demo and trial history is visible in the deal detail view and in the contact timeline.
