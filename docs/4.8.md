Based on section 4.8 of the specs — **Contracts & Legal** — here is the fully revised version adjusted for Cloudflare R2, the `AppServiceProvider` extension, and the correct Flysystem setup.

## Feature 1: Contract template library with clause management

**Story:** As an admin, I can build and maintain a library of contract templates composed of reusable clauses so that agents generate consistent, legally sound contracts without starting from scratch.

Acceptance criteria:
- An admin can create a named contract template with a description, contract type (MSA, NDA, SLA, renewal, upsell, or custom), and an active or inactive status.
- A template is composed of an ordered list of clauses. Clauses are added from the clause library or created inline. Clause order within a template is configurable via drag-and-drop.
- The clause library is a standalone repository of reusable clause records. Each clause has a name, body (rich text), category tag, and an active or inactive status.
- Clause bodies support variable placeholders. Supported variables are: contact full name, contact email, account name, account billing address, deal value, deal currency, agent full name, agent title, contract start date, contract end date, and any active custom contact or account fields.
- A clause can be marked as mandatory within a specific template. Mandatory clauses cannot be removed when an agent generates a contract from that template.
- A clause can be marked as optional. Optional clauses are presented to the agent at generation time with a toggle to include or exclude them.
- Template and clause editing is restricted to admin role. Managers can view templates and clauses but cannot modify them.
- Inactive templates do not appear in the contract generation flow. Inactive clauses within an active template are excluded from generation automatically.
- All template and clause changes are versioned. Each save creates a new version record. An admin can view version history and restore a previous version, which creates a new draft version from the restored content.
- Template and clause records are soft-deletable. Deleted templates and clauses remain linked to existing contracts that were generated from them but are removed from the generation flow.

## Feature 2: Contract generation from template

**Story:** As an agent, I can generate a contract from a template directly from a deal or account so that contract creation is fast, pre-populated with CRM data, and consistent with approved templates.

Acceptance criteria:
- Contract generation is initiated from the deal detail view or the account detail view via a generate contract action.
- The agent selects a template from the active template library. Only templates with active status are shown. Templates are filterable by contract type.
- After template selection, the system displays a variable fill screen. All variables that can be auto-filled from the linked deal, contact, and account are pre-populated. Variables with no resolvable source are presented as editable fields requiring manual input.
- Optional clauses are presented as toggles the agent can include or exclude. Mandatory clauses are shown as non-toggleable.
- The agent can preview the rendered contract before generating. The preview renders as a paginated document view showing the full contract with all variables resolved and selected clauses included.
- On confirmation, the system generates a PDF using DomPDF and stores it on the configured R2 disk using the path `contracts/{account_id}/{contract_id}/v{version_number}.pdf`. A contract record is created with status draft linked to the source deal, contact, and account.
- The generated PDF filename follows the naming convention: `ContractType_AccountName_YYYY-MM-DD.pdf` as the S3 object metadata display name, while the actual storage key follows the structured path above.
- Generation is processed synchronously for templates up to 20 pages. Templates expected to exceed 20 pages are processed as a queued job on the `default` Horizon queue. The agent receives an in-app notification via Reverb WebSocket when the PDF is ready.
- The agent can regenerate a contract in draft status. Regeneration creates a new PDF version, increments the version number on the contract record, stores the new file at the versioned R2 key, and retains all previous versions at their original keys.
- Contract generation requires the `contracts.create` permission, assigned to agent role and above by default.

## Feature 3: PDF generation and R2 storage

**Story:** As an agent, I can download, preview, and share contract PDFs that are securely stored on Cloudflare R2 so that documents are always accessible, never publicly exposed, and served via short-lived signed URLs.

Acceptance criteria:
- The R2 disk is registered in `config/filesystems.php` as a custom disk named `r2` using the `Storage::extend` driver registered in `AppServiceProvider`. The disk uses the following configuration:

```php
'r2' => [
    'driver'                  => 's3',
    'key'                     => env('R2_ACCESS_KEY_ID'),
    'secret'                  => env('R2_SECRET_ACCESS_KEY'),
    'region'                  => 'auto',
    'bucket'                  => env('R2_BUCKET'),
    'endpoint'                => env('R2_ENDPOINT'),
    'use_path_style_endpoint' => false,
    'url'                     => env('R2_URL'),
    'visibility'              => 'private',
],
```

- The `AppServiceProvider` boot method registers the `r2` driver extension wiring the AWS S3 client directly to the R2 endpoint so that `Storage::disk('r2')->temporaryUrl()` works correctly:

```php
Storage::extend('r2', function ($app, $config) {
    $client = new S3Client([
        'credentials' => [
            'key'    => $config['key'],
            'secret' => $config['secret'],
        ],
        'region'                  => $config['region'],
        'version'                 => 'latest',
        'endpoint'                => $config['endpoint'],
        'use_path_style_endpoint' => $config['use_path_style_endpoint'] ?? false,
    ]);

    $adapter = new AwsS3V3Adapter($client, $config['bucket']);

    return new FilesystemAdapter(
        new Filesystem($adapter, $config),
        $adapter,
        $config,
        $client
    );
});
```

- The `.env` file for the dev environment defines the following variables. All are required for the R2 disk to initialise correctly:

```env
FILESYSTEM_DISK=r2
R2_ACCESS_KEY_ID=your_r2_access_key_id
R2_SECRET_ACCESS_KEY=your_r2_secret_access_key
R2_BUCKET=your-bucket-name
R2_ENDPOINT=https://<your_account_id>.r2.cloudflarestorage.com
R2_URL=
```

- All contract PDFs are stored on the R2 disk with `visibility` set to `private`. The bucket is not publicly accessible. No file is ever stored with public visibility.
- PDF preview and download are served exclusively via signed URLs generated using `Storage::disk('r2')->temporaryUrl($path, now()->addMinutes(15))`. Signed URLs expire after 15 minutes. Requesting a document after expiry generates a fresh signed URL — the expired URL itself returns a 403.
- The contract detail view fetches a fresh signed URL on page load and renders the PDF inline in a browser PDF viewer. The signed URL is not cached client-side.
- A download action triggers a direct signed URL download. The download event is logged in the audit trail with the user ID, timestamp, and contract ID.
- All PDF versions of a contract are stored at versioned R2 keys: `contracts/{account_id}/{contract_id}/v{version_number}.pdf`. The contract detail view shows a version history list with version number, generation timestamp, and generating agent per version.
- The active version is the most recently generated PDF. An agent can designate a previous version as active from the version history panel. The change is logged in the audit trail.
- Contracts linked to soft-deleted contacts or accounts retain their R2 files. Files are only deleted when an admin explicitly purges a contract record, requiring a secondary confirmation dialog. Purge deletes all versioned keys for that contract from R2.
- PDF file size is validated after generation before the upload to R2 is attempted. Contracts exceeding 50MB trigger an admin alert, the generation job is marked failed with a descriptive error, and no file is written to R2.
- `phpunit.xml` sets `FILESYSTEM_DISK=local` so all tests use the local disk driver. No real R2 operations are performed during test runs:

```xml
<env name="FILESYSTEM_DISK" value="local"/>
```

## Feature 4: E-signature workflow

**Story:** As an agent, I can send a contract for e-signature and track signing status per party so that contracts are executed digitally without physical paperwork.

Acceptance criteria:
- E-signature is initiated from a contract record in draft or sent status. The agent selects the signature provider (DocuSign or the built-in OTP signing flow) in the send for signature screen.
- The agent specifies one or more signatories. Each signatory requires a name, email address, and role (counterparty, witness, or approver). Signing order is configurable — sequential (each party signs after the previous) or parallel (all parties receive the request simultaneously).
- On send, the contract status advances to sent. A signatory record is created per party with status pending.
- For DocuSign or HelloSign: the CRM creates an envelope via the provider API, attaches the active PDF fetched from R2 via a signed URL, assigns signatories, and stores the external envelope ID on the contract record. Provider webhook callbacks update each signatory's status (viewed, signed, declined) in real time.
- For the built-in OTP flow: each signatory receives an email with a unique signing link. The link opens a hosted signing page that fetches the contract PDF from R2 via a freshly generated signed URL valid for the duration of the signing session (60 minutes). The signatory enters their name, checks a consent checkbox, and enters a one-time password sent to their email or phone. On valid OTP entry, the signing is recorded with timestamp, IP address, and user agent. The OTP expires after 10 minutes.
- When all signatories have signed, the contract status automatically advances to signed. The deal owner and contract creator receive an in-app notification via Reverb WebSocket and an email notification.
- If any signatory declines to sign, the contract status moves to declined. The contract creator receives an immediate notification with the declining party's name and any reason provided.
- The contract detail view shows a signatory status panel listing each party, their role, status (pending, viewed, signed, declined), and timestamp for each status change.
- A signed contract is locked. Its PDF stored on R2 cannot be replaced or overwritten. Any amendments require creating a new contract record with a new generation and a new R2 storage path.
- The e-signature send action requires the `contracts.sign` permission, assigned to agent role and above by default.

## Feature 5: Contract status lifecycle

**Story:** As an agent, I can track a contract through its full lifecycle from draft to expiry so that no contract falls through the cracks and renewals are managed proactively.

Acceptance criteria:
- Contracts follow this status lifecycle: Draft → Sent → Signed → Active → Expiring → Expired. A contract can also move to Declined (from Sent) or Terminated (from Active or Signed).
- Status transitions are constrained. A contract can only move forward in the lifecycle via defined transitions. Invalid transitions are rejected with a validation error.
- The Draft → Sent transition is triggered by the e-signature send action. The Sent → Signed transition is triggered automatically when all signatories have signed. The Signed → Active transition is triggered automatically when the contract start date is reached, evaluated by the nightly scheduler command `php artisan contracts:activate`.
- The Active → Expiring transition is triggered automatically by the nightly scheduler when the contract end date is within the configured expiry warning window (default 90 days). The Expiring → Expired transition fires on the day after the contract end date.
- A manager or admin can manually trigger the Active → Terminated transition. Termination requires a mandatory termination reason and effective date. Terminated contracts are flagged with the termination reason in the detail view.
- Each status transition is written to the audit trail via Spatie Activitylog with the actor (user or system), timestamp, old status, and new status.
- The contract list view shows a status badge per contract with colour coding: draft (gray), sent (blue), signed (teal), active (green), expiring (amber), expired (red), declined (coral), terminated (dark gray).
- A contract in any status can be duplicated by a manager to create a new draft contract pre-populated with the same template, clauses, and variable values. The duplicate is created with status draft, no signatory records, and no R2 files — a new PDF must be generated for the duplicate.

## Feature 6: Renewal reminders

**Story:** As an account manager, I can receive automated renewal reminder notifications as a contract approaches expiry so that I can act on renewals before the contract lapses.

Acceptance criteria:
- Renewal reminders are sent at three configurable intervals before a contract's end date. Default intervals are 90 days, 60 days, and 30 days. Intervals are configurable per contract type in system settings.
- Reminders are evaluated and dispatched by the nightly scheduler via `php artisan contracts:reminders`. On each run the command identifies contracts whose end date falls exactly at a reminder interval from the current date and dispatches a `SendContractRenewalReminder` queued job on the `default` Horizon queue.
- Each reminder is sent via in-app notification through the Reverb WebSocket channel and email to the contract's assigned account manager. If no account manager is assigned, the reminder goes to the admin.
- The reminder email includes: contract name, account name, contract value, start date, end date, days remaining, and a direct link to the contract detail view. The PDF is not attached to the reminder email — the agent accesses it via the signed URL in the CRM.
- A reminder log is maintained on the contract record showing the date, interval (90/60/30 day), and delivery status (delivered, failed) of each reminder sent.
- A manager can manually suppress future reminders for a specific contract by toggling a suppress reminders flag on the contract. Suppression is logged in the audit trail with the suppressing user and a required reason.
- If a contract moves to Terminated or Expired status before a scheduled reminder, that reminder job and all subsequent reminder jobs are cancelled by checking contract status at job execution time before sending.
- An account manager can configure their personal reminder preferences (in-app only, email only, or both) from their notification preferences panel.
- A renewal pipeline view lists all contracts in Expiring status sorted by end date ascending, showing days remaining, contract value, account name, and assigned account manager.

## Feature 7: Contract performance tracking

**Story:** As an account manager, I can track milestones and KPIs against an active contract so that I can monitor whether contractual commitments are being met on both sides.

Acceptance criteria:
- A contract in Active or Signed status can have one or more milestone records added to it. Each milestone requires a name, due date, and assigned party (our organisation or the counterparty).
- Each milestone has a status of pending, completed, or missed. The assigned account manager updates the status manually. Completion requires a completion note and the actual completion date.
- A milestone whose due date passes without being marked completed is automatically set to missed status by the nightly scheduler command `php artisan contracts:milestones`. The assigned account manager receives an in-app Reverb notification when a milestone is marked missed.
- KPI fields are configurable per contract type by an admin. KPI field types are numeric (with target value and actual value), percentage, and date. Agents update KPI actual values from the contract detail view.
- A KPI whose actual value falls below the configured threshold is highlighted in amber on the contract detail view.
- The contract performance panel shows all milestones in a checklist view and all KPIs in a key-value table. Overall performance is summarised as a percentage of milestones completed on time.
- Milestone and KPI updates are written to the audit trail via Spatie Activitylog.
- Contract performance data is included in a contract analytics report in the analytics module showing milestone completion rates and KPI achievement rates across all active contracts.
- Performance tracking fields are visible to the assigned account manager, their manager, and admins. Agents without account manager access to the contract cannot view performance data.

## Feature 8: Contract repository

**Story:** As an agent or manager, I can search, filter, and access all contracts in a central repository so that any contract is retrievable in seconds with its PDF accessible via a secure signed URL.

Acceptance criteria:
- The contract repository is a searchable, filterable list of all contract records the current user has permission to access. Full-text search via Meilisearch through Laravel Scout covers contract title, account name, contact name, and clause content. The `Contract` model implements the `Searchable` trait and is indexed on creation and update.
- The repository supports filtering by: status, contract type, account, assigned account manager, start date range, end date range, and contract value range. Filters are applied via Spatie Laravel Query Builder.
- Each row in the repository list shows: contract title, account name, contract type, status badge, value, start date, end date, days remaining or days since expiry, and assigned account manager.
- The list is sortable by all displayed columns. Default sort is end date ascending.
- Clicking a contract row opens the contract detail view. A preview panel option fetches a fresh R2 signed URL and renders the active PDF version inline without leaving the repository list.
- Bulk actions available to managers and admins: bulk export selected contracts as a ZIP of PDFs, bulk update assigned account manager, and bulk add a tag. The ZIP export generates each PDF's signed URL from R2, downloads the file server-side, packages them into a ZIP, stores the ZIP temporarily on the `local` disk, and provides a one-time download link that expires after 30 minutes.
- Contracts are tagged with free-text labels. Tags are searchable and filterable in the repository.
- The repository paginates at 25 rows per page with an option to increase to 50 or 100.
- Access is scoped by permission. Agents see contracts linked to their assigned accounts and contacts. Managers see their team's contracts. Admins see all contracts.

## Feature 9: Legal matter log

**Story:** As an agent or manager, I can record legal matters linked to contacts and accounts so that legal correspondence, disputes, and case notes are tracked within the CRM.

Acceptance criteria:
- A legal matter record can be created from a contact detail view or an account detail view. Required fields are subject and status. Optional fields are description (rich text), matter type (dispute, correspondence, regulatory, advisory, or custom), assigned legal contact (a CRM user), and linked contract.
- Matter status lifecycle: open → in progress → pending external → resolved → closed. Status transitions are manually updated by the assigned user or a manager.
- Each legal matter has a notes thread where agents append timestamped case notes in rich text. Notes are append-only — no editing or deletion is permitted after posting. This produces an immutable case record.
- File attachments on matters and notes are stored on the R2 disk using the path `legal/{matter_id}/attachments/{filename}`. Access is via 15-minute signed URLs identical to the contract PDF access pattern. No attachment is ever publicly accessible.
- Attachment uploads are handled via Spatie Media Library configured to use the `r2` disk. The media library `disk` setting in `config/media-library.php` is set to `r2`:

```php
'disk_name' => env('MEDIA_DISK', 'r2'),
```

- With a corresponding `.env` entry:

```env
MEDIA_DISK=r2
```

- And in `phpunit.xml` for tests:

```xml
<env name="MEDIA_DISK" value="local"/>
```

- A matter can be linked to one or more contracts. Linked contracts are shown in the matter detail view. Clicking a linked contract navigates to the contract detail view.
- A matter can be linked to one or more contacts and accounts. Matters appear in the linked contact's and account's detail view under a legal matters panel.
- Access requires the `legal.view` permission. Creating and editing requires `legal.manage`. Both are assigned to manager and admin roles by default.
- All matter records are written to the audit trail. Status changes, note additions, and attachment uploads are all logged via Spatie Activitylog.
- Legal matters are excluded from all customer-facing views and never appear in the self-service portal.
- A legal matter summary report in the analytics module is accessible to compliance officers and admins, showing open matter count by type, average resolution time, and matters by assigned legal contact.
