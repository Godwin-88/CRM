Based on section 4.7 of the specs — **Analytics & Intelligence** — here are the user stories and acceptance criteria for each listed feature.

---

## Feature 1: CRM dashboard

**Story:** As an agent or manager, I can view a role-appropriate CRM dashboard on login so that I have an immediate overview of my pipeline, activity, and key metrics without navigating to individual modules.

Acceptance criteria:
- The dashboard is the default landing page after login. Its content is scoped by role: agents see their own metrics, managers see their team's metrics, admins see organisation-wide metrics.
- Agent dashboard displays: open deal count and total value, activities due today and overdue, open ticket count, recent interactions (last 5), and a mini pipeline view showing deals by stage.
- Manager dashboard displays: team pipeline value by stage, team activity completion rate, team ticket SLA breach count, top 5 deals by value, and agent performance summary (tickets resolved, deals moved this week per agent).
- Admin dashboard displays: organisation-wide pipeline value, total contacts created this month, campaign sends this month, revenue closed this month, system health indicators (queue depth, failed jobs, last scheduler run), and active user count.
- Each metric tile is a link that navigates to the relevant filtered view — clicking open ticket count navigates to the ticket list filtered to open tickets assigned to the current user.
- Dashboard widgets are configurable per user. Each user can add, remove, and reorder widgets from a widget library. Configuration is persisted per user.
- Dashboard data is cached for 15 minutes per user. A manual refresh button forces a cache bust and reloads all widgets.
- The dashboard does not update in real time except for the activity feed widget, which receives WebSocket push updates for new assignments and mentions.
- Dashboard layout is responsive and renders correctly on tablet-sized screens (minimum 768px width) as well as desktop.

## Feature 2: Sales forecasting

**Story:** As a sales manager, I can view a sales forecast combining weighted pipeline data and historical win rates so that I can project revenue accurately and identify gaps against targets.

Acceptance criteria:
- The forecast view shows three figures side by side for the selected period: target revenue (manually entered per period by a manager), weighted pipeline value (sum of deal value × stage probability for deals with close dates in the period), and best-case pipeline value (sum of all open deal values regardless of probability).
- Period selector supports: current month, next month, current quarter, next quarter, and custom date range.
- Historical win rate per pipeline stage is calculated from closed deals in the trailing 12 months and shown in a reference table alongside the configured probability for each stage.
- A deal-level breakdown table lists every open deal contributing to the forecast for the selected period, showing deal name, account, stage, value, weighted value, owner, and expected close date. The table is sortable and filterable by owner and stage.
- A forecast gap indicator shows the difference between the target and the weighted pipeline value. The indicator is colour-coded: green if weighted value meets or exceeds target, amber if within 20% below target, red if more than 20% below target.
- Forecast data is recalculated on page load. It is not cached.
- A manager can enter or update the revenue target for any period directly from the forecast view. Targets are stored per team per period.
- Forecast is accessible to managers and admins. Agents can view a personal forecast scoped to their own deals but cannot see targets.
- Forecast data is available via the analytics API endpoint for export to external BI tools.

## Feature 3: Customer analytics

**Story:** As a customer experience manager, I can view customer analytics covering segment performance, cohort retention, and churn risk so that I can understand the health of the customer base and act on trends.

Acceptance criteria:
- The customer analytics view has three tabs: segment performance, cohort retention, and churn risk.
- Segment performance tab shows for each active segment: contact count, average CLV, average deal value, average CSAT score, open ticket count, and campaign engagement rate (open rate of last campaign sent to the segment).
- Cohort retention tab groups contacts by the month they were first created (acquisition cohort) and shows what percentage of each cohort remains active (defined as having at least one interaction) at 1, 3, 6, and 12 months. Results are displayed as a cohort grid table.
- Churn risk tab lists all contacts with a high churn risk score, showing their name, account, assigned agent, days since last interaction, open ticket count, last NPS score, and churn risk score. The list is sortable by churn risk score descending.
- From the churn risk list, a manager can bulk-enrol selected contacts into a reactivation sequence with a single action.
- All three tabs support filtering by contact type, loyalty tier, owner, and date range.
- Analytics data on all tabs reflects the state as of the last nightly analytics job. The last calculation timestamp is shown on each tab.
- Each tab's data is exportable as CSV individually.

## Feature 4: Growth analytics

**Story:** As a sales manager, I can view growth analytics covering lead conversion, customer acquisition cost, and LTV to CAC ratio so that I can assess the efficiency of the sales and marketing investment.

Acceptance criteria:
- The growth analytics dashboard shows: total leads created in the period, lead-to-opportunity conversion rate, opportunity-to-closed-won conversion rate, and end-to-end lead-to-customer conversion rate.
- Customer Acquisition Cost (CAC) is calculated as total campaign spend in the period divided by number of new customers (contacts moved to customer type) in the period. Campaign spend is entered manually per campaign by a manager. CAC is shown per month and as a rolling 3-month average.
- LTV to CAC ratio is calculated using the predicted LTV from the CLV module divided by the current CAC. A ratio below 3:1 is highlighted in amber, below 1:1 in red.
- A conversion funnel chart visualises the drop-off at each pipeline stage across all deals created in the selected period.
- Time-to-close is shown as average days from deal creation to closed won for the selected period, broken down by pipeline and by deal value band.
- All metrics are filterable by pipeline, owner, team, and date range.
- Growth analytics data is calculated nightly. The last calculation timestamp is displayed.
- The dashboard is accessible to managers and admins only.
- All metrics on the dashboard are exportable as a single CSV report.

## Feature 5: Finance analytics

**Story:** As a finance manager, I can view revenue analytics covering product performance, revenue allocation, and accounts receivable aging so that I can monitor financial health and support reporting.

Acceptance criteria:
- The finance analytics dashboard shows: total revenue closed in the period (sum of closed won deal values), revenue by product or product category, revenue by account, revenue by agent, and month-over-month revenue trend as a line chart.
- Revenue allocation shows how closed won revenue is distributed across pipelines, contact types, and regions (derived from account billing address country).
- Accounts receivable aging shows outstanding invoices bucketed by age: current (0–30 days), 31–60 days, 61–90 days, and over 90 days. Each bucket shows total value and count of invoices.
- Clicking an aging bucket opens a filtered list of the invoices in that bucket showing account name, invoice value, due date, and assigned agent.
- All finance metrics are filterable by date range, pipeline, product category, and account.
- Finance analytics data is calculated nightly. The last calculation timestamp is displayed.
- Revenue data shown in the finance dashboard is sourced from closed won deals and linked invoice records. It does not pull from the accounting integration directly — it reflects CRM-recorded data only.
- Access to the finance analytics dashboard requires the `analytics.finance` permission, assigned to finance manager and admin roles by default.
- All metrics are exportable as CSV and as a formatted PDF report via the report builder.

## Feature 6: MarCom analytics

**Story:** As a marketing manager, I can view aggregated marketing and communications analytics across all campaigns and channels so that I can assess overall marketing performance and channel effectiveness.

Acceptance criteria:
- The MarCom analytics dashboard shows aggregate metrics across all campaigns in the selected period: total sends by channel, overall open rate, overall click rate, overall conversion rate, total unsubscribes, and total revenue attributed to campaigns.
- Channel performance comparison shows email, SMS, push, and in-app metrics side by side so the manager can compare channel effectiveness at a glance.
- Campaign ROI is calculated per campaign as revenue attributed divided by campaign spend. Campaigns without spend entered show revenue attributed only, with ROI marked as not set.
- A top campaigns table ranks campaigns by revenue attributed, open rate, and click rate. The ranking dimension is selectable.
- Mobile analytics shows push notification delivery rate, open rate, and click rate separately from email metrics.
- Attribution model is last-click by default. A configurable option per dashboard view allows switching to first-click attribution. The active attribution model is labelled clearly on the dashboard.
- All MarCom analytics data is filterable by campaign tag, channel, segment, and date range.
- Analytics data reflects the state as of the last nightly calculation. The timestamp is displayed.
- The dashboard is accessible to marketing managers and admins. Agents do not have access.
- Full dashboard data is exportable as CSV in a single export action.

## Feature 7: Compliance and GRC intelligence

**Story:** As a compliance officer, I can view audit trail reports and access anomaly detection alerts so that I can demonstrate regulatory compliance and respond to suspicious access patterns.

Acceptance criteria:
- The compliance dashboard shows: total audit log entries in the period, breakdown by event type (created, updated, deleted, viewed, exported), top 10 users by audit event count, and a timeline chart of audit activity by day.
- Audit trail reports are filterable by user, event type, model type (contact, deal, contract, etc.), date range, and IP address.
- Report results are paginated at 50 rows per page and exportable as CSV. The export includes all filter parameters in the file header for traceability.
- Access anomaly detection flags the following patterns as anomalies: a single user generating more than 500 audit events in one hour, a bulk export of more than 1,000 contact records, login from an IP address not seen in the last 30 days for that user, and more than 5 failed login attempts within 10 minutes for any account.
- Detected anomalies are surfaced in an anomaly alerts panel on the compliance dashboard. Each alert shows the user, event type, timestamp, and a brief description of why it was flagged.
- An anomaly alert can be acknowledged by a compliance officer or admin, adding an acknowledgement note and timestamp. Unacknowledged alerts older than 24 hours trigger an email notification to the admin.
- Audit logs are retained for 7 years. The retention period is configurable by an admin. Logs older than the retention period are purged by a scheduled nightly command.
- The compliance dashboard and audit export require the `compliance.view` permission, assigned to compliance officer and admin roles by default.
- Audit log records are immutable. No user, including admins, can edit or delete individual audit log entries through the CRM interface.

## Feature 8: Predictive deal scoring

**Story:** As a sales manager, I can view a predictive score on each deal indicating its likelihood of closing so that agents focus effort on the deals most likely to convert.

Acceptance criteria:
- Every open deal displays a predictive score from 0 to 100 and a corresponding label: cold (0–25), warm (26–50), hot (51–75), and very hot (76–100).
- The score is calculated by a rule-based scoring model using the following signals: days in current stage vs historical average days in that stage, number of interactions in the last 14 days, whether a demo or trial has been completed, deal value relative to average deal value for the pipeline, contact engagement score, and days until expected close date.
- Each signal contributes a weighted points value. Signal weights are configurable by an admin in the scoring settings panel without requiring a code deployment.
- The score is recalculated whenever any of its contributing signals change (deal stage move, new interaction logged, trial completed, close date updated) as a queued job.
- A nightly job recalculates all open deal scores to catch time-based signal decay (e.g. days in stage increasing passively).
- Deal score is visible on the deal card in the Kanban board, in the deal list view, and in the deal detail view.
- The deal list view is sortable by score descending so agents can work top-down through their highest-priority deals.
- Score history is not stored per deal — only the current score and its last calculated timestamp are persisted.
- A manager can override the score for a specific deal with a manual score and a reason note. Overridden scores are visually indicated. The override is cleared automatically when the deal moves to a new stage.

## Feature 9: Exploratory analysis

**Story:** As a manager or analyst, I can perform ad-hoc exploratory analysis on any CRM entity by applying filters, groupings, and aggregations so that I can answer one-off business questions without waiting for a pre-built report.

Acceptance criteria:
- The exploratory analysis tool is accessible from the analytics module. The user selects an entity type to analyse: contacts, accounts, deals, interactions, tickets, campaigns, or contracts.
- For the selected entity, the user can apply one or more filters using the same filter rule builder as the segmentation engine. Filters support all standard and custom fields for the entity type.
- The user can group results by one dimension (e.g. group deals by stage, or contacts by type). Grouping shows the count and an optional aggregate (sum, average, min, max) of a numeric field per group.
- Results are displayed as a table and as a bar or pie chart depending on the group dimension. The user can toggle between table and chart view.
- Results can be exported as CSV. The export applies all active filters and groupings.
- A filter and grouping configuration can be saved as a named report. Saved reports appear in the report library.
- Exploratory analysis queries run synchronously for result sets under 10,000 rows. Queries expected to exceed 10,000 rows are processed as background jobs and the user is notified when results are ready.
- The tool is accessible to manager and admin roles. Agents do not have access to exploratory analysis.
- No raw SQL input is permitted. All queries are constructed through the filter and grouping UI and executed via Eloquent query builder only.

## Feature 10: Report builder and scheduled delivery

**Story:** As a manager, I can build custom reports, save them, and schedule them for regular email delivery so that key metrics reach the right people automatically without manual exports.

Acceptance criteria:
- A report is built by selecting an entity type, applying filters, selecting fields to display, choosing a sort order, and optionally applying a grouping and aggregation. This mirrors the exploratory analysis tool but with the addition of field selection and scheduling.
- Reports support a tabular output format and an optional summary chart (bar, line, or pie) based on the grouping dimension.
- A saved report has a name, description, owner, and visibility setting (private — visible to owner only, or shared — visible to all managers and admins).
- Shared reports appear in a team report library. Any manager or admin can run a shared report. Only the report owner or an admin can edit or delete it.
- A report can be scheduled for recurring delivery. Supported frequencies are: daily, weekly (day of week configurable), and monthly (day of month configurable).
- Scheduled delivery sends the report as a CSV attachment and an inline summary table to a configurable list of email recipients. Recipients do not need to be CRM users.
- Each scheduled delivery run is logged with the timestamp, recipient list, and row count. Failed deliveries are retried once after 30 minutes and the owner is notified of a persistent failure.
- A report can be run on demand at any time from the report library. On-demand runs for large result sets (over 10,000 rows) are processed as background jobs with an in-app notification on completion.
- Reports are exportable as CSV or as a formatted PDF. PDF output uses a standard report template with the CRM logo, report name, filter summary, and generation timestamp in the header.
- The report builder and library are accessible to manager and admin roles. Agents cannot create or view reports.
