Based on section 4.10 of the specs — **Security & Access Control** — here are the user stories and acceptance criteria for each listed feature.

---

## Feature 1: Role-Based Access Control

**Story:** As an admin, I can assign roles to users and define granular permissions per role so that access to CRM data and actions is controlled consistently across the application.

Acceptance criteria:
- The system uses Spatie Laravel Permission. Five default roles are seeded on installation: Admin, Manager, Agent, Read-Only, and API Client.
- Permissions are organised by module and action, following the pattern `{module}.{action}` (e.g. `contacts.create`, `contracts.sign`, `analytics.finance`). Each permission referenced throughout the specs (e.g. `contacts.export`, `loyalty.adjust`, `legal.manage`) is seeded as a discrete permission record.
- A role can be assigned one or more permissions via a permission matrix UI in the admin settings panel. The matrix groups permissions by module with checkboxes, and supports a select-all toggle per module.
- A user can be assigned one or more roles. Effective permissions are the union of all permissions across the user's assigned roles.
- Custom roles can be created by an admin in addition to the five defaults. A custom role starts with zero permissions and is configured via the permission matrix.
- The Admin role cannot be deleted and always retains all permissions. Attempting to remove a permission from the Admin role or delete the Admin role returns a validation error.
- Role and permission changes take effect immediately for the affected user's next request — no re-login is required. This is achieved by clearing the user's permission cache on role or permission assignment change.
- All role assignments and permission changes are written to the audit trail with the acting admin, the affected user, and the before and after state.
- The API Client role is intended for OAuth2 application tokens (Laravel Passport) and is not assignable to interactive user accounts through the standard user management UI — it is assigned automatically to Passport client credential tokens.

## Feature 2: Multi-Factor Authentication

**Story:** As a user, I can enable two-factor authentication on my account so that my account is protected even if my password is compromised.

Acceptance criteria:
- MFA is implemented using TOTP (Time-based One-Time Password) via `pragmarx/google2fa-laravel`. Setup generates a QR code the user scans with an authenticator app (Google Authenticator, Authy, or similar).
- During MFA setup, the user must enter a valid code from their authenticator app to confirm the setup before MFA is activated on the account. An invalid confirmation code does not activate MFA.
- On activation, the system generates 10 single-use recovery codes displayed once to the user with a prompt to save them securely. Recovery codes are stored hashed in the database.
- MFA is mandatory for users with the Admin or Manager role, as configured by `MFA_REQUIRED_ROLES=admin,manager` in the environment configuration. Users with these roles cannot access the CRM beyond the MFA setup screen until MFA is activated.
- MFA is optional for Agent and Read-Only roles by default, but an admin can change the `MFA_REQUIRED_ROLES` configuration to mandate it for any role.
- At login, a user with MFA enabled is prompted for their TOTP code after successful password authentication. Three consecutive failed TOTP attempts trigger a 5-minute lockout on TOTP entry for that account, with the lockout event logged to the audit trail and a security notification emailed to the user.
- A user can use a recovery code instead of a TOTP code at login. Each recovery code can only be used once. After use, the remaining valid recovery code count is shown to the user, with a prompt to regenerate codes if fewer than 3 remain.
- A user can disable MFA from their security settings, requiring re-entry of their password and a valid TOTP code or recovery code to confirm. Disabling MFA on an Admin or Manager account is blocked while `MFA_REQUIRED_ROLES` includes their role — the user is shown an explanation and directed to contact another admin if MFA must be removed.
- An admin can reset (force-disable) MFA for another user in the event of lost device access. This action requires the admin's own MFA confirmation and is logged in the audit trail. The affected user is notified by email and prompted to set up MFA again on next login if their role requires it.

## Feature 3: Single Sign-On

**Story:** As a user, I can log in using my organisation's Google Workspace or Azure AD account so that I use existing corporate credentials without managing a separate CRM password.

Acceptance criteria:
- SSO is implemented via Laravel Socialite, supporting Google Workspace (OAuth2/OpenID Connect) and Azure AD (OAuth2/OpenID Connect, configurable for SAML via a provider package if required).
- SSO providers are configured per organisation in admin settings with client ID, client secret, and tenant/domain restrictions. A domain restriction (e.g. `@company.com`) ensures only accounts from the configured domain can authenticate via that provider.
- On first SSO login, if no CRM user account exists with the matching email address, account creation behaviour is configurable: auto-create a new user with a default role (e.g. Agent), or block login and require an admin to pre-provision the account. Default behaviour is block — auto-creation is opt-in per organisation.
- If a CRM user account exists with a matching email and the account currently uses password authentication, the first successful SSO login links the SSO identity to the existing account. The user retains the ability to authenticate via password unless an admin disables password login for SSO-linked accounts.
- SSO login bypasses the local MFA prompt if the identity provider enforces its own MFA, configurable per provider. If local MFA enforcement is enabled regardless of SSO, the user completes the local MFA step after SSO authentication succeeds.
- A user can view and unlink their connected SSO identity from their security settings, provided they have a password set on their account or another authentication method available. Unlinking the only available authentication method is blocked.
- SSO authentication events (success, failure, new account creation, identity linking) are written to the audit trail with the provider name, email, and outcome.
- If SSO is disabled or misconfigured (expired credentials, provider outage), users with password authentication available can still log in normally. The SSO option is hidden from the login screen automatically if the provider configuration test fails, with an admin alert generated.

## Feature 4: Privileged session management

**Story:** As an admin, I can perform destructive or highly sensitive operations only after re-confirming my identity in a privileged session so that accidental or unauthorised destructive actions are minimised.

Acceptance criteria:
- A set of actions are designated as privileged: permanently purging soft-deleted records, modifying role permissions, disconnecting integrations, force-resetting another user's MFA, bulk-deleting contacts or accounts, and rotating API keys.
- Attempting a privileged action when the user's session is not in privileged mode prompts a re-authentication modal requiring the user's password and, if MFA is enabled, a valid TOTP or recovery code.
- On successful re-authentication, the session enters privileged mode for 15 minutes. Privileged mode is indicated by a persistent visual badge in the application header.
- All privileged actions performed while in privileged mode are written to the audit trail with a `privileged_session: true` flag, the re-authentication timestamp, and the action performed.
- Privileged mode expires automatically after 15 minutes of being entered, regardless of activity. Attempting a privileged action after expiry re-triggers the re-authentication modal.
- A user can manually exit privileged mode at any time via a button in the header badge, immediately requiring re-authentication for any further privileged action.
- Privileged mode does not extend the user's overall session lifetime — it is a sub-state of an existing authenticated session and does not bypass the standard session timeout.
- If a user's account does not have MFA enabled, privileged mode re-authentication uses password confirmation only. This is logged distinctly in the audit trail (`privileged_session_mfa: false`) so admins can identify privileged actions performed without MFA-backed confirmation.

## Feature 5: Data classification

**Story:** As an admin, I can classify fields by sensitivity level so that access to PII, financial, and confidential data is controlled independently of general record access.

Acceptance criteria:
- Three sensitivity tags are supported: PII (personally identifiable information), Financial, and Confidential. Tags are applied at the field level via a configuration map maintained by admins, covering both standard and custom fields.
- Standard field classifications are pre-seeded based on the data model: contact email, phone, and national ID fields are tagged PII; invoice amounts, bank details, and vendor financial fields are tagged Financial; legal matter contents and banking relationship notes are tagged Confidential.
- A user without the corresponding sensitivity permission (`data.view_pii`, `data.view_financial`, `data.view_confidential`) sees classified fields masked in the UI — displayed as a partial mask (e.g. `j***@*****.com` for email, `***-***-1234` for phone) rather than fully hidden, so the user knows the field exists without exposing its value.
- API responses apply the same masking rules based on the requesting token's associated user permissions. Masking is applied server-side in API resource transformers, never client-side.
- An admin can grant temporary elevated access to view a classified field for a specific record to a specific user, with an expiry time (maximum 24 hours) and a mandatory business justification note. This grant is logged in the audit trail and automatically revoked at expiry.
- Bulk exports (CSV, reports) apply the same masking rules. A user without `data.view_pii` who exports a contact list receives masked PII fields in the export file — the export does not bypass field-level classification.
- Custom fields can be tagged with a sensitivity level by an admin when the field is created or edited. Sensitivity tagging is optional for custom fields and defaults to no classification (visible to all users with general record access).
- Changing a field's sensitivity classification is itself a privileged action requiring privileged session mode, and is written to the audit trail with the old and new classification.

## Feature 6: GDPR / PDPA compliance

**Story:** As a data protection officer, I can manage data subject requests and ensure the CRM supports the right to erasure, data portability, and consent tracking so that the organisation meets GDPR and Kenya's Data Protection Act obligations.

Acceptance criteria:
- Each contact record includes consent fields: `marketing_consent` (boolean), `data_processing_consent` (boolean), and `consent_timestamp` (datetime, recorded when consent is given or withdrawn). Consent state changes are logged in the audit trail.
- A data subject request (DSR) module allows an admin or compliance officer to initiate a request of type: access (export all data), erasure (right to be forgotten), or rectification, against a specific contact by email or contact ID.
- An access request generates a structured export (JSON and human-readable PDF) of all data held about the contact across all modules — contact and account records, interactions, deals, tickets, contracts, invoices, loyalty transactions, survey responses, and audit log entries referencing that contact. The export is stored on R2 at `dsr/{request_id}/export.zip` with a 7-day signed URL provided to the requesting officer.
- An erasure request, on confirmation, anonymises the contact's PII fields (name replaced with a hash-derived placeholder, email and phone nulled or replaced with anonymised values, custom PII-tagged fields cleared) while retaining non-PII transactional data required for legal, financial, or audit purposes (e.g. invoice totals and dates remain, but the customer name on historical invoices becomes "Anonymised Customer"). The anonymisation logic is encapsulated in the `AnonymizeDeletedContacts` console command, runnable on-demand for a specific contact ID in addition to its daily scheduled run for soft-deleted contacts past the retention grace period.
- The retention grace period before automatic anonymisation of soft-deleted contacts is configurable by an admin (default 30 days from soft-deletion). During the grace period, a soft-deleted contact can be restored with all data intact. After anonymisation, restoration is not possible — anonymisation is irreversible.
- A DSR record tracks: request type, requesting contact's identifier, date received, date completed, the handling officer, and the outcome. DSR records themselves are retained for 7 years as evidence of compliance, even after the underlying contact data has been anonymised.
- Erasure requests are blocked if the contact has an active contract, an unresolved legal matter, or an outstanding invoice balance greater than zero. The blocking condition is shown to the requesting officer with a link to the blocking record. The officer can override the block with a privileged session confirmation and a mandatory justification note, which is logged.
- Initiating and completing DSR requests requires the `dsr.manage` permission, assigned to admin and compliance officer roles by default.

## Feature 7: Field-level audit trail for sensitive fields

**Story:** As a compliance officer, I can view a field-level history of changes to sensitive data so that I can investigate who changed what and when for any PII or financial field.

Acceptance criteria:
- Fields tagged PII or Financial (per Feature 5's classification map) have field-level change tracking enabled via Spatie Activitylog's attribute logging, capturing the old value, new value, changing user, timestamp, and IP address for every update.
- Field-level audit entries for PII and Financial fields store the old and new values in encrypted form in the `audit_logs` table, decrypted only for display to users with the corresponding `data.view_pii` or `data.view_financial` permission.
- A field history view is accessible from any record's detail view for users with appropriate permissions, showing a chronological list of changes to that record's sensitive fields, filterable by field name and date range.
- Bulk operations (imports, merges, anonymisation) that affect sensitive fields generate a single audit log entry per affected record per field, not a single aggregate entry for the entire bulk operation — ensuring per-record traceability.
- Field-level audit entries are immutable and excluded from the standard audit log retention purge if they relate to an open DSR request or active legal matter, overriding the default 7-year retention if a longer retention is legally required for that specific record.
- Viewing field-level sensitive audit history is itself logged as an access event (without recursive logging of the access-log-viewing event itself, to prevent infinite logging loops).
- The field history view requires the `compliance.field_audit` permission, assigned to compliance officer and admin roles by default.

## Feature 8: Security event logging

**Story:** As a security administrator, I can view a log of security-relevant events so that I can monitor for suspicious activity and respond to incidents.

Acceptance criteria:
- The following events are logged to a dedicated security events table (distinct from the general `audit_logs` table, though both feed the compliance dashboard): login success, login failure, password change, password reset request, MFA enabled/disabled, MFA challenge failure, privilege change, privileged session entry, API token created/revoked, SSO identity linked/unlinked, bulk export performed, and account lockout.
- Each security event record includes: event type, user ID (nullable for failed logins with unknown email), IP address, user agent, timestamp, and outcome (success/failure).
- Login failures are recorded even for email addresses that do not correspond to an existing account, without revealing in the system's response whether the account exists — but the internal log records the attempted email for pattern analysis.
- A security events dashboard, accessible to security administrators and admins, shows recent events in a filterable, searchable table (by event type, user, IP address, and date range) and a summary panel highlighting: failed login attempts in the last 24 hours, new IP addresses used by existing users in the last 7 days, and accounts with MFA disabled among Admin/Manager role users (flagging non-compliance with `MFA_REQUIRED_ROLES`).
- Account lockout triggers automatically after 5 consecutive failed login attempts for the same account within 10 minutes, locking the account for 15 minutes. A lockout event is logged and the account holder receives an email notification with guidance, without including a reset link in the notification itself (to avoid the notification being usable as an attack vector if intercepted).
- Security events are retained for the same period as general audit logs (7 years, configurable) and are included in the nightly retention purge command alongside `audit_logs`.
- Security event log export (for SIEM integration or incident investigation) is available as CSV or JSON to security administrators and admins, with the export action itself logged as a security event.
- Access to the security events dashboard requires the `security.events` permission, assigned to security administrator and admin roles by default.

## Feature 9: Rate limiting on API endpoints

**Story:** As an admin, I can configure rate limits on API endpoints so that the CRM is protected from abuse, accidental overload, and brute-force attacks.

Acceptance criteria:
- Rate limiting uses Laravel's built-in rate limiter backed by Redis. Default limits are defined per route group: authentication endpoints (login, password reset) at 5 requests per minute per IP, general API endpoints at 60 requests per minute per authenticated token, and bulk operation endpoints (import, export, bulk update) at 10 requests per minute per token.
- Rate limit configuration per route group is adjustable by an admin via a settings panel without requiring a code deployment, stored in a `rate_limits` configuration table read by the rate limiter middleware at request time (with a short cache TTL to avoid a database query per request).
- Requests exceeding the rate limit receive a 429 response with a `Retry-After` header indicating seconds until the limit resets, and a response body explaining the limit that was exceeded.
- Rate limit state is tracked in Redis using a sliding window approach, keyed by IP address for unauthenticated endpoints and by API token for authenticated endpoints.
- The IVR ingestion endpoint (200 requests per minute per source IP) and the kiosk ingestion endpoint (500 requests per minute, triggering an admin alert above this threshold) referenced in section 4.3 use this same rate limiting infrastructure with their endpoint-specific overrides.
- Per-client (Passport OAuth2) rate limit overrides can be configured for trusted third-party integrations that require higher throughput than the default, set individually per OAuth client by an admin.
- Repeated rate limit breaches (more than 20 occurrences within 1 hour from the same IP or token) generate a security event of type `rate_limit_abuse` visible on the security events dashboard, and trigger an automatic temporary block (1 hour) of that IP or token at the gateway level.
- Rate limit headers (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`) are included in all API responses, including successful ones, so well-behaved clients can self-throttle proactively.

## Feature 10: Input sanitization and CSRF/XSS protection

**Story:** As a developer, I need the application to enforce consistent input sanitization and CSRF/XSS protections so that the CRM is protected against injection and cross-site attacks by default rather than relying on per-feature implementation.

Acceptance criteria:
- All Blade templates use Laravel's default `{{ }}` escaping for output. Raw output via `{!! !!}` is restricted to a documented allowlist of fields known to contain rich text (knowledge base articles, email templates, canned responses, contract clauses, legal matter notes) and is passed through a server-side HTML sanitizer (e.g. HTML Purifier) before storage, not only at output time.
- The HTML sanitizer allowlist permits: headings, paragraphs, lists, links (with `rel="noopener noreferrer"` automatically added to external links), bold, italic, underline, tables, and images with `src` restricted to the application's R2 domain and approved CDN domains. Script tags, inline event handlers (`onclick`, etc.), `javascript:` URIs, and `<iframe>` (except for an allowlisted set of approved embed domains for the knowledge base) are stripped on save.
- CSRF protection uses Laravel's default `VerifyCsrfToken` middleware for all session-based (web) routes. API routes authenticated via Sanctum tokens or Passport OAuth2 are exempt from CSRF (token-based auth is the CSRF mitigation for stateless requests), but session-based SPA requests using Sanctum's SPA authentication mode retain CSRF protection via the `XSRF-TOKEN` cookie.
- All React/Vue frontend components rendering user-generated content that has passed through the rich-text allowlist use a client-side sanitizer (DOMPurify) as a defence-in-depth layer before rendering with `v-html` or `dangerouslySetInnerHTML`, even though the content was sanitized server-side on save.
- File upload validation enforces MIME type checking against file content (not just file extension) using Laravel's `mimes` and a server-side magic-byte check. Uploaded files are scanned by an antivirus stub/hook point (`FileUploadScanner` interface) before being persisted to R2, with the hook point ready for integration with a virus scanning service.
- All Form Request validation classes apply input length limits, type coercion, and format validation (email format, phone format per E.164, date format) before any controller logic executes. No raw `$request->all()` is mass-assigned to a model without passing through a Form Request's validated array.
- SQL injection prevention is enforced structurally: all database queries use Eloquent ORM or the query builder with parameter binding. Raw queries (`DB::raw`, `whereRaw`) are restricted to a documented list of locations, each requiring a code review comment justifying the raw query and confirming no user input is interpolated into the raw string.
- A Content Security Policy (CSP) header is set on all responses, restricting script sources to the application's own domain and the CDN allowlist (cdnjs.cloudflare.com and similar), restricting `frame-src` to the approved embed domains, and disallowing inline scripts except where a nonce is generated per request for required inline scripts.

## Feature 11: Password policy enforcement

**Story:** As an admin, I can enforce password complexity and history rules so that user accounts meet a baseline security standard against credential-based attacks.

Acceptance criteria:
- Password policy is configured via Laravel's `Password` validation rule combinator, enforcing: minimum 12 characters, at least one uppercase letter, one lowercase letter, one number, and one symbol, by default. Minimum length is configurable by an admin (cannot be set below 10).
- New passwords are checked against the "Have I Been Pwned" compromised password database via Laravel's `Password::uncompromised()` rule. A password found in a known breach is rejected with a message advising the user to choose a different password, without revealing the specific breach source.
- Password history is enforced: a user cannot reuse any of their last 5 passwords. The last 5 password hashes (not plaintext) are retained per user for comparison on password change.
- Password expiry is configurable per role (default: no expiry for Agent and Read-Only, 90-day expiry for Admin and Manager roles). Users with an expiring password receive an in-app warning 7 days before expiry. On expiry, the user is required to set a new password before accessing any other part of the application.
- Account lockout after 5 failed login attempts (per Feature 8) is independent of and in addition to password policy — password policy governs password creation/change, lockout governs login attempts.
- Password reset tokens (via Laravel's standard password reset flow) expire after 60 minutes and are single-use. Requesting a new reset token invalidates any previously issued unused tokens for that account.
- All password changes (self-service and admin-initiated) trigger an email notification to the account's registered email address informing them of the change, without including the new password, and providing guidance to contact support if the change was not initiated by them.
- Password policy configuration changes (minimum length, expiry periods, history depth) are privileged actions requiring privileged session mode and are written to the audit trail.
