# Section 4.13: Platform Documentation & Help Center

## Overview

This module delivers in-product documentation covering every feature specified in sections 4.1–4.12. Unlike the README (developer-facing, lives in the repo, covers setup/architecture), this is **user-facing**, lives inside the CRM application itself, and is structured around *what a user is trying to do* rather than the technical module breakdown. Three audiences need fundamentally different entry points: agents (task-oriented, "how do I..."), managers/admins (configuration-oriented, "how do I set up..."), and developers/integrators (reference-oriented, API and webhook docs — which already exist as the developer portal from section 4.11 Feature 8 and should be linked, not duplicated).

---

## Feature 1: Documentation content model and authoring

**Story:** As a documentation maintainer, I can author and organize help articles that map to CRM features so that documentation stays structured, searchable, and tied to the modules it describes.

Acceptance criteria:
- Documentation articles use the same content model as knowledge base articles (section 4.6 Feature 4) — title, body (rich text with the same sanitization rules per section 4.10 Feature 10), category, author, and the same draft → in review → approved → published → archived lifecycle from content lifecycle management (section 4.4 Feature 10).
- Documentation articles are distinguished from customer-facing knowledge base articles by an `audience` field: `agent`, `manager`, `admin`, or `all`. This determines visibility in the in-app help panel (Feature 2) but does not affect the underlying article storage or versioning mechanism — both article types share the same table and editorial workflow.
- Each article can be tagged with one or more `feature_refs` corresponding to spec section numbers (e.g. `4.2.2` for Kanban board, `4.8.4` for e-signature). This tagging powers the contextual help mapping in Feature 2.
- Articles support embedded screenshots and short looping video clips (MP4, max 10MB) stored on R2 at `docs/{article_id}/media/{filename}`, served via signed URLs consistent with every other R2-backed feature in the specs.
- Articles are versioned identically to knowledge base articles (section 4.6 Feature 4) — each published update creates a new version, with history and restore available to admins.
- A "last verified" date is tracked per article, set manually by the author when they confirm the article still matches current behaviour. Articles not verified within 6 months are flagged in the documentation maintainer dashboard as potentially stale.
- Article authoring requires the `docs.manage` permission, assigned to admin role by default and extendable to designated documentation owners per team.

## Feature 2: Contextual in-app help

**Story:** As an agent, I can access help relevant to the screen I'm currently on without leaving my workflow so that I get unblocked quickly during real tasks.

Acceptance criteria:
- A persistent help icon is available in the application header on every screen. Clicking it opens a slide-over panel (not a new tab) showing articles relevant to the current screen, determined by matching the current route to `feature_refs` tags (Feature 1).
- The contextual panel shows up to 5 most relevant articles for the current screen, ranked by `feature_refs` match specificity (an article tagged with the exact sub-feature ranks above one tagged with the parent section only).
- If no contextually relevant article exists for a screen, the panel falls back to a search box and a list of the most-viewed articles in the current module's parent category (e.g. on any Contracts screen, fall back to top Contracts articles generally).
- The panel includes a persistent search box at the top, searching all published articles the current user's `audience` permits via Meilisearch/Scout, consistent with the knowledge base search pattern (section 4.6 Feature 4).
- Clicking an article opens it within the same slide-over panel (not a page navigation), preserving the user's place in their current task. A "open in full documentation view" link is available for articles the user wants to read at length or bookmark.
- The contextual panel records which article was viewed from which screen (route + article ID), feeding the "most viewed per module" fallback ranking and the documentation maintainer's coverage gap report (Feature 5).
- For managers and admins, the contextual panel additionally surfaces configuration-audience articles relevant to settings screens — e.g. on the loyalty program configuration screen (section 4.5 Feature 1), the panel shows the admin-audience article on configuring tiers and points rules, not the agent-audience article on viewing a contact's loyalty balance.
- The panel is dismissible and its open/closed state does not persist across page navigations — it always starts closed on a new screen, avoiding a help panel that follows the user around intrusively.

## Feature 3: Full documentation centre

**Story:** As any user, I can browse a complete, organized documentation centre so that I can learn the platform broadly, not just get unstuck on one screen.

Acceptance criteria:
- The documentation centre is accessible from the application header (alongside the contextual help icon from Feature 2) as a dedicated full-page view, and is also reachable from the notification centre's "notification preferences" pattern — i.e. a persistent, predictable location rather than buried in settings.
- The centre's navigation structure mirrors the spec's module organization (sections 4.1–4.12) but uses task-oriented language rather than spec section numbers — e.g. "Managing contacts and accounts" (4.1), "Running your sales pipeline" (4.2), "Customer conversations" (4.3), through to "Working together" (4.12). Spec section numbers are not exposed to end users; they exist only in the `feature_refs` metadata (Feature 1).
- Each top-level section has a landing page with a brief overview and a list of articles within it, grouped by sub-topic. The grouping mirrors the "Feature N" breakdown from each spec section but, again, in plain language.
- The centre's content is filtered by the viewing user's role-derived `audience` — an agent does not see admin-configuration articles in the main navigation, but can still find them via search if they have the underlying permission (e.g. a team lead with partial admin permissions per section 4.12 Feature 3).
- A "Getting started" section is pinned at the top of the centre's navigation regardless of role, containing a short onboarding path: first login, navigating the dashboard (section 4.7 Feature 1), understanding notifications (section 4.12 Feature 7), and where to get help — this is the only section guaranteed to be read by brand-new users and should not assume prior context.
- Search within the full documentation centre is the same Meilisearch/Scout-backed search as the contextual panel (Feature 2), but presented as a full results page rather than an inline panel, with filters for category and content type (article vs video).
- Breadcrumb navigation shows the user's path within the documentation hierarchy (e.g. Documentation > Sales pipeline > Deal stage automations) and each breadcrumb level is clickable.
- The documentation centre is responsive and meets the same WCAG 2.1 AA standard required of the self-service portal (section 4.5 Feature 7).

## Feature 4: Role-based onboarding checklists

**Story:** As a new user, I can follow a role-specific checklist of first tasks so that I learn the platform through guided action rather than passive reading.

Acceptance criteria:
- On first login, a user is shown a dismissible onboarding checklist scoped to their assigned role(s) — separate checklists exist for Agent, Manager, and Admin, with a user holding multiple roles seeing the union of relevant items, de-duplicated.
- Agent checklist items include: complete profile and set language preference (section 4.3 Feature 10), enable MFA if not already required (section 4.10 Feature 2), review the unified interaction inbox (section 4.3 Feature 1), open a deal on the Kanban board (section 4.2 Feature 2), and view the notification centre (section 4.12 Feature 7).
- Manager checklist items include all agent items plus: configure or review a pipeline (section 4.2 Feature 1), review the agent performance dashboard (section 4.6 Feature 7), and review the team management screen (section 4.12 Feature 3).
- Admin checklist items include all manager items plus: review the RBAC permission matrix (section 4.10 Feature 1), connect at least one integration from the connector marketplace (section 4.11 Feature 6), and review the compliance dashboard (section 4.7 Feature 7).
- Each checklist item links directly to the relevant screen and, where applicable, opens the contextual help panel (Feature 2) for that screen automatically on arrival, with a "mark complete" action that the user triggers manually (completion is not auto-detected from system activity, to avoid false completions from incidental navigation).
- Checklist progress persists per user. The checklist can be reopened at any time from the documentation centre's "Getting started" section (Feature 3) even after all items are completed, but is no longer shown automatically on login once 100% complete or explicitly dismissed.
- A manager can view (read-only) the onboarding checklist completion status of their direct team members from the team management screen (section 4.12 Feature 3), as a lightweight onboarding-progress indicator — not a performance metric, and explicitly excluded from the agent performance dashboard (section 4.6 Feature 7) to avoid conflating learning progress with work output.

## Feature 5: Documentation coverage and feedback loop

**Story:** As a documentation maintainer, I can see where documentation is missing, outdated, or unhelpful so that the help content stays aligned with the platform as it evolves.

Acceptance criteria:
- Every article (Feature 1) has a "was this helpful" yes/no prompt at its conclusion, consistent with the knowledge base helpful-vote mechanism (section 4.6 Feature 4). An article with a helpful ratio below 40% (minimum 10 votes) is flagged for review, identical to the knowledge base threshold.
- A documentation maintainer dashboard, accessible to users with `docs.manage`, shows: articles flagged for low helpfulness, articles not verified within 6 months (Feature 1), and a coverage gap report.
- The coverage gap report cross-references `feature_refs` tags (Feature 1) against the full list of spec features (4.1 through 4.12, each numbered feature within each section) and lists any feature with zero associated articles — giving the maintainer a checklist of documentation debt against the specification itself.
- The contextual help panel (Feature 2) includes a "this didn't help, request documentation" action when no relevant article is found for a screen (the fallback case). This creates a documentation request record containing the route/screen identifier, the requesting user, and an optional comment, visible in the maintainer dashboard's request queue.
- Documentation requests from the contextual panel are deduplicated — multiple requests for the same screen increment a request count rather than creating duplicate queue entries, helping the maintainer prioritize by demand.
- The maintainer dashboard shows the most-viewed articles and most-used search terms (including searches that returned zero results — a strong signal for missing content) over a configurable period, using the view tracking established in Feature 2.
- None of the feedback or analytics in this feature is visible to the agents/managers/admins consuming documentation — it is exclusively a maintainer-facing tool, keeping the end-user documentation experience free of meta-commentary about its own completeness.

## UX summary across the audiences

For an agent mid-task, the experience is: hit a wall on a screen, click the persistent header help icon, get a short ranked list of relevant articles in a slide-over that doesn't lose their place, read or watch, close the panel, continue working. They never need to know the documentation centre exists unless they want to browse.

For a manager or admin doing configuration, the same contextual entry point surfaces configuration-audience content automatically based on the screen, and the onboarding checklist (Feature 4) gives them a guided first pass through the admin-heavy setup tasks (RBAC, integrations, compliance) that the agent checklist never shows.

For a documentation maintainer, the loop closes through Feature 5: low-helpfulness flags, staleness flags, coverage gaps against the spec itself, and a request queue from real users hitting real walls — all without that feedback layer leaking into the end-user experience.

For a developer or integrator, this module deliberately does not duplicate the OpenAPI developer portal (section 4.11 Feature 8) — the documentation centre's navigation can link out to `/docs` for API reference, but API documentation generation, versioning, and the "try it out" sandbox remain owned by that feature, not rebuilt here.
