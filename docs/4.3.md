Based on section 4.3 of the README — **Customer Interactions & Omni-Channel** — here are the user stories and acceptance criteria for each listed feature.

---

## Feature 1: Unified interaction inbox

**Story:** As an agent, I can view all customer interactions across every channel in a single inbox so that I have complete context without switching between tools.

Acceptance criteria:
- The inbox displays interactions of type call, email, chat, SMS, in-person, field visit, kiosk, WhatsApp, Facebook, LinkedIn, Instagram, and TikTok, all in a single chronological feed.
- Each entry shows the channel type with a distinct icon, direction (inbound/outbound), contact name, subject or summary, date, and assigned agent.
- The inbox can be filtered by channel type, direction, date range, agent, and contact without leaving the view.
- Unread or unactioned interactions are visually distinguished from reviewed ones. An agent can mark an interaction as reviewed.
- Clicking an interaction opens a detail panel showing the full body, attachments, linked deal or ticket, and the contact's last three interactions for quick context.
- The inbox supports both an all-interactions view (team-wide, for managers) and a personal view scoped to the current agent's assigned contacts.
- The inbox paginates at 50 items per page and supports infinite scroll as an alternative rendering mode.
- New inbound interactions appear in the inbox in real time via WebSocket without requiring a page refresh.

## Feature 2: Email integration — inbound and outbound

**Story:** As an agent, I can send and receive emails from within the CRM so that all email communication is captured against the correct contact automatically.

Acceptance criteria:
- Outbound email is sent via the configured mail provider (Mailgun or Postmark). The agent composes from a rich-text editor inside the CRM, selects the contact, and sends.
- Sent emails are logged as an outbound interaction against the contact and any linked deal or ticket immediately on send.
- Inbound email is received via IMAP polling or provider webhook. Each inbound message is parsed for sender address and matched to an existing contact by email. If matched, an inbound interaction is created and linked to that contact.
- If the sender address does not match any contact, the email is placed in an unmatched inbox queue for manual review. An agent can assign it to an existing contact or create a new contact from it.
- Replies to a CRM-sent email are threaded under the original interaction using the Message-ID header.
- Email attachments are stored in S3 and linked to the interaction record. Supported attachment types are PDF, DOCX, XLSX, PNG, JPG up to 25MB total per email.
- An agent can use saved email templates when composing. Templates support variable placeholders for contact name, agent name, and account name.
- Inbound email processing is handled as a queued job. Processing delays exceeding 5 minutes trigger an alert to the admin.

## Feature 3: Call logging — manual and CTI

**Story:** As an agent, I can log calls against a contact manually or have them captured automatically via CTI integration so that all voice interactions are on record.

Acceptance criteria:
- Manual call log requires contact, direction (inbound/outbound), date, and duration. Outcome and notes are optional.
- A logged call creates an interaction record of type call linked to the contact and optionally to a deal or ticket selected by the agent.
- CTI integration receives call events via Twilio or Vonage webhooks. On call completion, the system automatically creates a call interaction record with caller ID matched to a contact, direction, duration, and a recording URL if available.
- If a CTI inbound caller ID does not match any contact, the call is logged as an unmatched call record. The agent is notified and can assign it to a contact post-call.
- Call recordings stored via CTI are linked as attachments on the interaction record. Access to recordings requires the `interactions.recordings` permission.
- During an active CTI call, the agent's CRM screen displays the matched contact's name, last interaction summary, and open deals in a pop-up screen. This requires the Reverb WebSocket channel to be active.
- Call interaction records are included in the contact timeline and in the unified inbox.
- Call logs can be edited by the creating agent within 24 hours. After that they are locked and only a manager can edit.

## Feature 4: Live chat

**Story:** As an agent, I can respond to customer chat messages from within the CRM so that chat is handled alongside all other interaction types in one place.

Acceptance criteria:
- Inbound chat sessions are initiated from a CRM-hosted chat widget embedded on the customer's web property. The widget connects via Laravel Reverb WebSocket.
- A new chat session creates an interaction record of type chat, initially unassigned, and appears in the unified inbox with a visual indicator distinguishing it as live and pending.
- An agent accepts the chat session from the inbox. Acceptance assigns the session to that agent and notifies the customer that an agent has joined.
- Messages are exchanged in real time over WebSocket. Each message is persisted to the interaction body as an append-only transcript.
- If no agent accepts a chat within a configurable timeout (default 3 minutes), an automated message is sent to the customer and the session is queued for callback follow-up.
- At the start of a chat, the system attempts to identify the customer by a cookie or email address entered in the pre-chat form. If matched, the agent sees the contact record linked to the session.
- On chat close, the agent selects an outcome and optionally links the session to a deal or ticket. The transcript is stored as the interaction body.
- Concurrent chat sessions per agent are capped at a configurable limit (default 3). The system does not route new sessions to agents already at their cap.

## Feature 5: SMS via Twilio and Africa's Talking

**Story:** As an agent, I can send and receive SMS messages to contacts from within the CRM so that text communication is tracked alongside other channels.

Acceptance criteria:
- Outbound SMS is sent via the configured provider (Twilio or Africa's Talking, selectable per region in system settings). The agent composes from the contact detail view or from the interaction creation form.
- SMS character count and segment count are displayed in real time as the agent types. Messages exceeding one segment show a warning before sending.
- Sent SMS creates an outbound interaction record of type SMS linked to the contact.
- Inbound SMS is received via provider webhook. The system matches the sender phone number to a contact. If matched, an inbound SMS interaction is created and appears in the unified inbox.
- If the sender number does not match any contact, the SMS is placed in an unmatched queue identical to the email unmatched flow.
- SMS replies from a contact are threaded under the originating outbound message in the interaction detail view.
- Bulk SMS to a segment is handled through the campaign module, not the interaction inbox. Single ad-hoc SMS only is in scope for this feature.
- SMS delivery status (delivered, failed, pending) is updated via provider webhook callback and reflected on the interaction record.
- Africa's Talking is the default provider for KE region numbers. Twilio handles all other regions. Provider routing is automatic based on the contact's phone number country code.

## Feature 6: IVR call transcript ingestion

**Story:** As an agent, I can view IVR call transcripts linked to the relevant contact so that automated voice interactions are part of the complete customer history.

Acceptance criteria:
- IVR transcripts are ingested via a REST endpoint that accepts a JSON payload containing caller ID, call timestamp, duration, IVR path taken, and transcript text.
- The endpoint authenticates via HMAC signature verification using a shared secret configured per IVR system integration.
- On receipt, the system matches the caller ID to a contact by phone number. If matched, an inbound interaction of type IVR is created and linked to that contact.
- Unmatched IVR records are held in the unmatched queue and are visible to admins for manual assignment.
- The IVR path taken (menu selections and outcomes) is stored as structured JSON and rendered in a readable step-by-step format in the interaction detail view, not as raw JSON.
- IVR interactions appear in the contact timeline with a distinct IVR type indicator.
- The ingestion endpoint rate-limits to 200 requests per minute per source IP. Requests exceeding this return a 429 response.
- IVR transcript ingestion failures are logged and trigger an admin alert if more than 5 consecutive payloads from the same source fail validation.

## Feature 7: Field channel — mobile agent integration

**Story:** As a field agent, I can log interactions and access contact information from my mobile device while offline so that fieldwork is captured even without connectivity.

Acceptance criteria:
- The field agent mobile API supports reading contact details, account details, and scheduled activities while offline using a locally cached snapshot updated on last sync.
- The cache is populated when the agent opens the app with connectivity. The last sync timestamp is visible in the app.
- While offline, an agent can create call logs, visit notes, and activity completions. These are stored locally in a pending queue on the device.
- When connectivity is restored, the pending queue syncs automatically to the CRM. The agent receives a confirmation notification listing how many records were synced.
- Conflict resolution on sync: if a contact record was updated in the CRM while the agent was offline, the server version wins for shared fields. The agent's new interaction records are always accepted.
- Field visit interactions created via mobile appear in the unified inbox and the contact timeline with channel type field visit.
- The mobile API uses Bearer token authentication with a 30-day sliding expiry. Tokens are revocable by an admin from the security settings panel.
- Sensitive fields (national ID, financial data) are excluded from the offline cache regardless of what the agent has permission to view online.

## Feature 8: Kiosk interaction event ingestion

**Story:** As a CRM administrator, I can receive interaction events from self-service kiosks so that customer actions at kiosks are recorded against their contact record.

Acceptance criteria:
- Kiosk events are submitted to a dedicated REST endpoint accepting a JSON payload with kiosk ID, event type, customer identifier (phone, account number, or email), timestamp, and event metadata.
- The endpoint authenticates via API key scoped to the kiosk integration. Each physical kiosk or kiosk system has its own API key, registered in the integrations settings panel.
- The customer identifier in the payload is matched to a contact. If matched, an inbound interaction of type kiosk is created linked to that contact and the originating kiosk ID.
- If no match is found, the event is logged as an unmatched kiosk event in the admin queue.
- Kiosk interactions appear in the unified inbox and the contact timeline with a kiosk channel indicator.
- Event metadata is stored as structured JSON and rendered in human-readable format in the interaction detail view.
- Kiosk API keys can be rotated by an admin without downtime. The old key remains valid for 24 hours after rotation to allow kiosk systems to update.
- A kiosk event ingest rate exceeding 500 events per minute triggers a throttle response (429) and an admin alert.

## Feature 9: Contact centre queue stats dashboard

**Story:** As a contact centre supervisor, I can view real-time queue statistics so that I can monitor agent workload and respond to bottlenecks immediately.

Acceptance criteria:
- The dashboard displays: total open interactions, interactions by channel type, average wait time for unassigned interactions, interactions per agent, and SLA breach risk count.
- All metrics update in real time via WebSocket. The dashboard does not require manual refresh.
- Each metric tile shows the current value and a directional indicator (up/down) compared to the same metric 30 minutes ago.
- Clicking an agent's interaction count opens a filtered inbox view showing that agent's current open interactions.
- A supervisor can reassign an interaction to a different agent directly from the dashboard without opening the individual interaction record.
- The dashboard is scoped by team. A supervisor sees only the agents in their assigned team. Admins see all teams with a team selector dropdown.
- Historical queue data (last 7 days, hourly buckets) is available as a chart below the live stats panel for trend comparison.
- Access to the dashboard requires the `contact_centre.dashboard` permission, which is assigned to supervisor and admin roles by default.

## Feature 10: Multilingual agent interface

**Story:** As an agent, I can use the CRM interface in my preferred language so that I can work efficiently without language being a barrier.

Acceptance criteria:
- The CRM supports English (EN) and Swahili (SW) as the initial two interface languages.
- Language preference is set per user in their profile settings. The selected language applies to all UI labels, navigation, system messages, error messages, and notification text generated by the CRM.
- Switching language takes effect immediately on save without requiring a logout.
- Content entered by users (contact names, notes, interaction bodies) is stored and displayed as-entered regardless of the interface language — only system-generated UI text is translated.
- Automated email and SMS notifications sent to contacts use the contact's preferred language field if set, falling back to the system default language.
- Missing translation keys fall back to English. A missing translation never displays a raw translation key to the user.
- Adding a new language is supported via the translations configuration file. No code changes are required to add a language — only a new locale file and admin activation.
- The language selector is accessible from the login screen so agents can choose their language before authenticating.
