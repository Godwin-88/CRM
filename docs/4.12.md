Based on section 4.12 of the specs — **Workspace & Collaboration** — here are the user stories and acceptance criteria for each listed feature.

---

## Feature 1: Internal notes and comments (polymorphic)

**Story:** As an agent, I can add internal notes and comments to any CRM record so that context, discussion, and collaboration happen alongside the data they relate to.

Acceptance criteria:
- Comments are implemented as a polymorphic `comments` table (`commentable_type`, `commentable_id`), attachable to contacts, accounts, deals, tickets, contracts, campaigns, and any other model that includes the `HasComments` trait.
- A comment requires a body (rich text) and is associated with the authoring user and the commentable record. Empty comments cannot be submitted.
- Comments are displayed in a threaded panel on each record's detail view, ordered chronologically with newest at the bottom by default, with an option to reverse the order per user preference.
- Comments support the same rich-text sanitization rules as section 4.10 Feature 10 — HTML Purifier on save, DOMPurify on render, with the same allowlist (bold, italic, lists, links).
- A comment can be edited by its author within 15 minutes of posting (consistent with section 4.2 Feature 8's deal comment behaviour, now generalized across all entity types). After 15 minutes it is locked for editing.
- A comment can be deleted by its author or a manager. Deleted comments are replaced with a "comment deleted" placeholder showing the deletion timestamp and actor, never fully removed from the database.
- Internal comments never appear in the contact-facing timeline, self-service portal, or any customer-visible view, regardless of the entity type they're attached to.
- A record's detail view shows a comment count badge. The badge updates in real time via the Reverb WebSocket channel for that record when other users post comments while the page is open.

## Feature 2: @mention notifications

**Story:** As an agent, I can @mention a teammate in a comment or note so that they're notified directly and can act on the context I've raised.

Acceptance criteria:
- Typing `@` within any comment or note field opens an inline dropdown of team members, filtered by name as the user continues typing. The dropdown shows the user's name, role, and avatar.
- Selecting a user from the dropdown inserts a mention token into the comment body, rendered as a styled, clickable chip linking to that user's profile when the comment is displayed.
- On comment submission, each mentioned user receives an in-app notification (delivered via Reverb WebSocket if online) and an email notification, both linking directly to the record and scrolled to the specific comment.
- A user can disable email notifications for mentions independently of other notification types from their notification preferences panel (per section 4.4 Feature 1's variable system and consistent with existing preference patterns). In-app notifications cannot be fully disabled for mentions — only the email channel is configurable.
- A user can only be mentioned if they have at least view access to the record the comment is attached to. Users without access do not appear in the mention dropdown for that record, preventing notification of users about records they cannot open.
- The notification centre (Feature 7 below) aggregates all mention notifications and marks them as read when the user navigates to the linked comment, or can be marked read in bulk from the notification centre itself.
- Mentioning a user who is part of a team channel discussion (Feature 6 below) follows the same mechanism — mentions are mention-token based regardless of whether the comment is on a record or in a discussion board.
- Mention notification delivery (in-app and email) is processed via a queued job on the `default` Horizon queue, consistent with general notification handling described in section 4.5 Feature 3.

## Feature 3: Team management

**Story:** As an admin, I can organize users into teams so that records, dashboards, and reporting can be scoped to team boundaries.

Acceptance criteria:
- A team has a name, an optional description, and a designated team lead (a user with manager role or above). A user can belong to multiple teams, but has exactly one "primary team" used for default dashboard scoping.
- Team membership is managed by admins or the team lead from a team management screen, with add/remove member actions logged in the audit trail.
- Teams are used as the scoping mechanism referenced throughout the specs: the manager's pipeline forecast (section 4.2 Feature 4), the contact centre dashboard (section 4.3 Feature 9), the agent performance dashboard (section 4.6 Feature 7), and the CRM dashboard's manager view (section 4.7 Feature 1) — all scope "team" data based on team membership.
- A team can be archived (not deleted) when disbanded. Archived teams retain their historical association with records and reports for trend analysis, but no new members can be added and the team no longer appears in active team selectors.
- Each team has an activity feed — a chronological view of significant events involving any team member: deals closed, tickets resolved, contracts signed, and comments/mentions within team-visible records. This feed is the team-level complement to the personal activity feed on the user's dashboard.
- Team activity feed entries respect the same record-level access permissions as the underlying records — a team member without access to a specific deal does not see that deal's closure event in the feed, even though their teammate closed it.
- A user's removal from a team does not retroactively remove their historical contributions from that team's activity feed or reports — historical attribution is preserved.
- Team configuration (creation, archiving, membership changes) requires the `teams.manage` permission, assigned to admin role by default. Team leads have a scoped version of this permission limited to their own team's membership (add/remove members only, not create/archive teams).

## Feature 4: File attachments (polymorphic, S3/R2-backed)

**Story:** As an agent, I can attach files to any CRM record so that relevant documents, images, and exports are stored alongside the records they relate to.

Acceptance criteria:
- File attachments use Spatie Media Library configured to the `r2` disk (consistent with the configuration established in section 4.8's redevelopment), attachable to any model implementing `HasMedia` — contacts, accounts, deals, tickets, contracts (in addition to their dedicated PDF storage), and discussion board posts.
- Maximum file size per attachment is 25MB, enforced both client-side (immediate feedback before upload begins) and server-side (rejecting oversized uploads with a 422 and clear error message).
- Allowed file types are restricted to: PDF, DOCX, XLSX, PNG, JPG, CSV, and ZIP. MIME type validation follows the same magic-byte verification approach specified in section 4.10 Feature 10, not extension-based checking alone.
- Attachments are stored at the path `{model_type}/{model_id}/attachments/{filename}`, mirroring the structured key pattern established for contracts, invoices, and legal matters in sections 4.8 and 4.9.
- Attachment access is via 15-minute signed URLs generated through `Storage::disk('r2')->temporaryUrl()`, consistent with every other R2-backed feature. No attachment is ever publicly accessible.
- Each attachment shows: filename, file size, uploader, upload timestamp, and a download action. Image attachments (PNG, JPG) display an inline thumbnail generated server-side via Media Library's image conversions, with the thumbnail itself also served via signed URL.
- An attachment can be deleted by its uploader or a manager. Deletion removes the file from R2 (not soft-deleted at the storage layer) but the deletion event itself is recorded in the audit trail with the filename, uploader, and deleting user for traceability.
- Attachment uploads and deletions on any record are written to the audit trail via Spatie Activitylog, consistent with the audit pattern established for every other module.

## Feature 5: E-signature requests from within the CRM

**Story:** As an agent, I can initiate an e-signature request for any document — not just generated contracts — from within the CRM so that the e-signature workflow is available as a general collaboration tool, not limited to the contracts module.

Acceptance criteria:
- This feature generalizes the e-signature workflow established in section 4.8 Feature 4 (DocuSign/HelloSign or built-in OTP flow) to operate on any file attachment (Feature 4 above), not exclusively contract PDFs.
- An agent initiates an e-signature request from an attachment's context menu, selecting the signature provider, specifying signatories (name, email, role), and signing order — using the identical UI and backend flow as section 4.8 Feature 4.
- A generalized `signable` polymorphic relationship allows the e-signature/signatory tables from section 4.8 to attach to any model with file attachments, not only the `Contract` model — avoiding duplication of the signatory tracking schema.
- Signatory status tracking (pending, viewed, signed, declined), provider webhook handling, and the built-in OTP flow with 10-minute expiry all behave identically to section 4.8 Feature 4, regardless of which record type the signable attachment belongs to.
- When all signatories have signed, the signed document is stored as a new versioned attachment (incrementing the version per the existing attachment versioning convention) and the record's owner receives an in-app and email notification.
- A signed attachment, like a signed contract, is locked — it cannot be replaced or have its file deleted, only superseded by a new attachment version if a new signature round is initiated.
- E-signature requests on general attachments require the `attachments.sign` permission. By default this is granted to the same roles as `contracts.sign` (agent role and above), but can be restricted further per record type by an admin if needed.

## Feature 6: Internal discussion boards

**Story:** As a team member, I can participate in threaded discussions scoped to an account or deal so that broader strategic conversations have a home distinct from record-specific comments.

Acceptance criteria:
- A discussion board is created per account or per deal, accessible from a "Discussion" tab on that record's detail view. Unlike comments (Feature 1), which are typically short and reactive, discussion boards support longer-form threaded posts with titles.
- A discussion thread has a title, an originating post (rich text, same sanitization rules as comments), and an author. Replies to a thread are nested one level deep (replies to replies are flattened into the same level, avoiding deep nesting complexity in the UI).
- Threads can be pinned by a manager to keep important discussions at the top of the board. A maximum of 3 threads can be pinned simultaneously per board; pinning a 4th requires unpinning an existing one.
- Discussion board posts and replies support @mentions (Feature 2) and file attachments (Feature 4), using the same underlying mechanisms.
- A thread can be marked as resolved by its author or a manager, visually distinguishing it (e.g. greyed out, moved to a "resolved" filter view) without deleting it. Resolved threads can be reopened by the same roles.
- Discussion board content is included in the account or deal's activity feed entries (Feature 3's team activity feed) when a new thread is created — but individual replies do not generate separate activity feed entries, to avoid feed noise.
- Discussion boards are internal-only, consistent with comments (Feature 1) — never visible in the self-service portal or any customer-facing view.
- Search within a discussion board (title and body full-text) is available via the same Meilisearch/Scout infrastructure used elsewhere, scoped to threads and replies on that specific account or deal.

## Feature 7: Notification centre

**Story:** As a user, I can view all my notifications in a central notification centre so that mentions, assignments, reminders, and system alerts are accessible in one place regardless of which module generated them.

Acceptance criteria:
- The notification centre is accessible via a bell icon in the application header, showing an unread count badge. Clicking opens a dropdown panel (or full page on mobile) listing notifications in reverse chronological order.
- Notification types aggregated into the centre include (but are not limited to): @mentions (Feature 2), record assignments (deals, tickets, activities), SLA breach warnings (sections 4.5 and 4.6), contract renewal reminders (section 4.8), loyalty tier changes (section 4.5), milestone due/missed alerts (section 4.8), and security alerts relevant to the user (section 4.10).
- Each notification shows: an icon indicating its type/source module, a brief description, a relative timestamp, and is clickable to navigate directly to the relevant record or view.
- New notifications appear in real time via the Reverb WebSocket channel (`user.{userId}`) without requiring a page refresh, with a subtle visual indicator (e.g. badge pulse) for newly arrived items.
- Notifications can be marked as read individually, or all at once via a "mark all as read" action. Read notifications remain in the centre (not deleted) for a rolling 30-day history, after which they're purged by a nightly scheduled command.
- A "notification preferences" link within the centre navigates directly to the user's notification preferences panel (referenced throughout the specs — section 4.4 Feature 1, section 4.5 Feature 3, section 4.8 Feature 6, Feature 2 above), allowing per-category channel configuration (in-app, email, both, or none where permitted).
- The notification centre supports filtering by category (mentions, assignments, alerts, reminders) via a tab or dropdown filter at the top of the panel.
- Notification centre data loads via API on initial page load (last 30 days, paginated), with subsequent updates arriving exclusively via WebSocket — avoiding polling.

## Feature 8: Shared team calendar

**Story:** As a team member, I can view a shared calendar of activities, meetings, and demos so that scheduling conflicts are visible and team availability is transparent.

Acceptance criteria:
- The shared calendar aggregates: activities (section 4.2 Feature 6) with a due date/time, demos and trials (section 4.2 Feature 9), onboarding workflow milestones (section 4.5 Feature 6), and contract renewal/milestone dates (section 4.8 Features 6 and 7) — presented in a unified month/week/day calendar view.
- Calendar entries are colour-coded by type (activity, demo/trial, onboarding, contract milestone), with a legend, consistent with the colour conventions established for status badges elsewhere in the specs (e.g. section 4.8 Feature 5's status colours).
- The calendar can be filtered by team (Feature 3), individual user, and entry type. A user always sees their own entries by default; team-scoped views require the user to be a member of that team or have manager-level access.
- Clicking a calendar entry opens a preview popover showing key details and a link to the full record (the deal, ticket, contract, etc.) without leaving the calendar view.
- An entry's date/time can be adjusted by dragging it to a new slot on the calendar (for activities and demos/trials, which are user-editable records) — this updates the underlying record via the same API used by the Kanban board's optimistic update pattern (section 4.2 Feature 2), with the same snap-back-on-failure behaviour.
- Contract milestones and renewal reminders (read-only system-calculated dates) are displayed but cannot be dragged or edited from the calendar — attempting to drag shows a tooltip explaining the date is system-managed and where to edit it (the contract record).
- The calendar supports an iCal feed export (per user, with a unique unguessable URL) for subscription in external calendar applications (Google Calendar, Outlook). The feed URL can be regenerated by the user if compromised, invalidating the previous URL.
- Real-time updates: if another team member creates or moves a calendar entry the current user can see (per team/permission scoping), the calendar updates live via the Reverb WebSocket channel without requiring a refresh.

## Note on scope overlap

A few items in this section deliberately reference and extend features already specified in earlier sections rather than duplicating them — most notably e-signature (Feature 5 generalizes 4.8 from the README Feature 4), comments (Feature 1 generalizes the deal-specific comments from 4.2 Feature 8), and notification preferences (Feature 7 ties together the preference panels scattered across 4.4, 4.5, 4.8, and this section). This is intentional: the underlying schema and services (signatory tracking, comment threading, notification dispatch) are built once and reused, rather than each module reimplementing its own version. Worth flagging to your implementation team early so the comments/notifications/e-signature services are built as shared, generic services from the start rather than retrofitted later.
