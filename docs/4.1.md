Based on section 4.1 of the specs — **Contact & Account Management** — here are the user stories and acceptance criteria for each listed feature.

---

## Feature 1: Full CRUD for contacts and accounts with custom fields

**Story:** As an agent, I can create, view, edit, and delete contacts and accounts so that I maintain accurate stakeholder records.

Acceptance criteria:
- Creating a contact requires first name, last name, email, and type (lead/prospect/customer/partner). All other fields are optional.
- Email must be unique across non-deleted contacts. Submitting a duplicate email returns a validation error identifying the conflict.
- Editing a contact saves only changed fields and writes an audit log entry with old and new values for each changed field.
- Deleting a contact performs a soft delete. The record is excluded from all list views and search results but remains in the database and is visible in the audit trail.
- Accounts support the same CRUD operations. An account can optionally reference a parent account for subsidiary relationships.
- Custom fields can be defined per entity type (contact or account) by an admin. Supported types are text, number, date, dropdown, and boolean.
- Custom field values are saved alongside the standard fields and appear in the contact/account detail view.
- All CRUD operations enforce RBAC — agents can create and edit, managers can delete, read-only users can only view.

## Feature 2: Dynamic segmentation engine

**Story:** As a marketing manager, I can build segments using filter rules so that I can target the right contacts for campaigns and outreach.

Acceptance criteria:
- A segment is defined by one or more filter rules joined by AND or OR logic.
- Supported filter dimensions include: contact type, loyalty tier, CLV score range, geographic field, source channel, owner, created date range, custom field values, and last interaction date.
- A preview of matching contacts is available before saving, showing count and a sample of up to 10 records.
- Segments are dynamic — the contact list is re-evaluated at query time, not stored as a static list.
- Segment contact counts are cached and refreshed by the hourly scheduler command. The cache timestamp is visible in the segment detail view.
- A segment can be assigned to a campaign. Only contacts matching the segment at dispatch time are included in the send.
- Agents can view segments but only managers can create, edit, or delete them.

---

## Feature 3: Duplicate detection and merge workflow

**Story:** As an agent, I can identify and merge duplicate contact records so that the CRM does not hold redundant data on the same person.

Acceptance criteria:
- When creating or importing a contact, the system checks for existing records with the same email address or the same combination of first name, last name, and phone number.
- If a potential duplicate is found, the user is shown the conflicting record(s) and can choose to proceed as a new record, cancel, or initiate a merge.
- The merge workflow presents both records side by side. For each field, the user selects which value to keep or types a new value.
- On merge confirmation, the chosen record is retained with the selected field values. The discarded record is soft-deleted.
- All related data (interactions, deals, tickets, activities, contracts) from the discarded record is re-linked to the surviving record before deletion.
- A merge event is written to the audit log, recording both record IDs and the user who performed the merge.
- Merge is restricted to manager role and above.

## Feature 4: Contact timeline

**Story:** As an agent, I can view a chronological timeline for any contact so that I have full context before an interaction.

Acceptance criteria:
- The timeline displays all interactions, activities, deals, tickets, contracts, and loyalty transactions linked to the contact, ordered by date descending.
- Each entry shows its type (with a distinct visual indicator), date, a short summary, and the agent who created it.
- The timeline is paginated or infinitely scrolled — it does not load all records at once.
- Filtering by type (e.g. show only interactions, or only deals) is available without leaving the timeline view.
- The timeline updates in real time when a new interaction or activity is logged while the page is open, using the WebSocket channel.
- Read-only users can view the timeline. No create or edit actions are available from the timeline itself.

## Feature 5: Bulk import and export

**Story:** As a sales manager, I can import contacts from a CSV or Excel file and export filtered lists so that I can migrate data and share it with other tools.

Acceptance criteria:
- Import supports CSV and Excel (.xlsx) files up to 10MB.
- After file upload, the user is presented with a field mapping step where each column in the file is mapped to a contact field (including custom fields). Unmapped columns are ignored.
- The system validates all rows before processing. Rows with missing required fields or duplicate emails are flagged with a row-level error report. Valid rows are still imported.
- Import is processed as a queued job. The user receives an in-app notification when it completes, including a summary of records created, skipped, and failed.
- Export applies the current active filters on the contacts list. The user can select which fields to include.
- Export is processed as a queued job for lists over 500 records. The user receives a download link via in-app notification when the file is ready. Small exports (≤500) download immediately.
- Import and export actions are logged in the audit trail with the user ID, timestamp, and record count.
- Import is restricted to manager role and above. Export requires the `contacts.export` permission.

## Feature 6: Contact scoring model

**Story:** As a sales manager, I can configure scoring rules so that contacts are automatically ranked by their likelihood to convert or their value to the business.

Acceptance criteria:
- Scoring rules are configurable by admins. Each rule specifies a field condition and a point value (positive or negative).
- Supported conditions include: contact type, source channel, number of interactions in the last 30 days, deal stage reached, loyalty tier, and custom field values.
- A contact's score is the sum of all matching rule values. Scores update whenever a contact's data changes in a way that affects a rule.
- The score is visible as a numeric value and a colour-coded indicator (e.g. green/amber/red based on configurable thresholds) on the contact card and list view.
- Score recalculation for all contacts runs as a queued job triggered by the daily scheduler. Individual contact scores update immediately when triggered by an event.
- Score history is not stored — only the current score is persisted. The rules that contributed to the score are derivable from the current rule configuration.
- Scoring rules can be disabled without deleting them. Disabled rules do not contribute to score calculation.

---

## Feature 7: Relationship mapping

**Story:** As an agent, I can see and navigate the relationships between a contact and related accounts, deals, and tickets so that I understand the full picture without manual searching.

Acceptance criteria:
- A contact detail view shows associated accounts (a contact can belong to multiple accounts), open and closed deals, open and resolved tickets, and active contracts in dedicated relationship panels.
- Each related record in a panel is a link that navigates to that record's detail view.
- An account detail view shows its primary contacts, subsidiary accounts, deals, tickets, and contracts.
- Relationships can be created and removed from either side — linking a contact to an account from the contact view or from the account view produces the same result.
- A contact can be designated as the primary contact for an account. Only one primary contact is allowed per account at a time.
- Removing a relationship does not delete either record. The link is simply dissolved.
- Relationship changes are written to the audit log.
