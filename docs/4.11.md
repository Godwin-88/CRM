Based on section 4.11 of the specs — **API & Integrations** — here are the user stories and acceptance criteria for each listed feature, including direct answers to your two questions at the end.

## Feature 1: RESTful API (versioned)

**Story:** As a developer integrating with the CRM, I can consume a versioned REST API so that my integration remains stable even as the CRM evolves.

Acceptance criteria:
- All API endpoints are namespaced under `/api/v1/`. The version prefix is mandatory — unversioned API routes are not exposed.
- API responses follow a consistent envelope: `{ "data": ..., "meta": {...}, "links": {...} }` for collections (using Laravel's resource pagination), and `{ "data": {...} }` for single resources, generated via Laravel API Resources.
- Error responses follow a consistent structure: `{ "error": { "code": "...", "message": "...", "details": {...} } }` with appropriate HTTP status codes (422 for validation, 403 for authorization, 404 for not found, 429 for rate limit, 500 for server error).
- A future `/api/v2/` namespace can be introduced without breaking `/api/v1/` consumers. Versioning is implemented at the route and controller level — v2 controllers can extend v1 controllers and override only changed methods.
- Deprecation of a v1 endpoint requires a minimum 6-month notice period communicated via a `Deprecation` HTTP header on affected responses and documented in the API changelog accessible from the developer portal.
- All list endpoints support pagination via `page` and `per_page` query parameters, with a maximum `per_page` of 100 enforced server-side regardless of what the client requests.
- All list endpoints support filtering and sorting via Spatie Laravel Query Builder conventions (`filter[field]=value`, `sort=field` or `sort=-field` for descending).
- API responses include a `X-Request-ID` header on every response, generated per request and included in any related log entries, to assist with support and debugging.

## Feature 2: API authentication

**Story:** As a developer, I can authenticate API requests using a bearer token or API key so that access to CRM data is secured and attributable to a specific client.

Acceptance criteria:
- Two authentication mechanisms are supported for `/api/v1/`: Sanctum personal access tokens (for first-party SPA and mobile clients) and Passport OAuth2 (for third-party integrations).
- A Sanctum token is created by an authenticated user from their security settings, with a configurable token name, optional ability scopes (e.g. `contacts:read`, `deals:write`), and an optional expiry date. Tokens without an expiry date are flagged with a warning recommending the user set one.
- Token abilities are enforced via Sanctum's `tokenCan()` checks in addition to standard RBAC — a token must both belong to a user with the relevant permission AND carry the relevant ability for a request to succeed. This allows a user to issue a read-only token even if their own role permits writes.
- API key authentication (a simpler alternative for service-to-service or webhook-receiving integrations such as the kiosk and IVR endpoints from section 4.3) is implemented via a custom header (`X-API-Key`), validated against hashed keys stored in the `integrations` table, scoped to a specific integration record and its designated permissions.
- All tokens and API keys are displayed in full only once at creation time. Subsequent views show only a masked identifier (e.g. last 4 characters) sufficient for the user to identify which token is which.
- A token or API key can be revoked at any time by the issuing user (for personal tokens) or an admin (for integration API keys). Revocation is immediate — the next request using the revoked credential returns a 401.
- All authentication failures (invalid token, expired token, revoked token, invalid API key) are logged as security events per section 4.10 Feature 8, including the masked credential identifier used.
- Token and API key creation, viewing the credential list, and revocation are accessible from a dedicated API access panel in user settings (for personal tokens) and the integrations settings panel (for service API keys).

## Feature 3: OAuth2 server for third-party integrations

**Story:** As a third-party developer, I can register an OAuth2 application and obtain tokens via the authorization code or client credentials flow so that my application can act on behalf of CRM users or as a service account.

Acceptance criteria:
- OAuth2 is implemented via Laravel Passport. An admin can register OAuth2 clients from the integrations settings panel, specifying client name, redirect URIs, and grant types permitted (authorization code, client credentials, or both).
- The authorization code flow presents an authorization screen to the CRM user, showing the requesting application's name, requested scopes in plain language (e.g. "Read your contacts", "Create and update deals"), and Allow/Deny actions.
- Granted authorization code tokens inherit the authorizing user's RBAC permissions intersected with the granted scopes — an OAuth token cannot exceed what the authorizing user could do themselves.
- Client credentials flow tokens are assigned the API Client role (per section 4.10 Feature 1) with permissions configured per registered client by an admin, independent of any individual user's permissions.
- Access tokens expire after 1 hour. Refresh tokens expire after 30 days and are single-use — using a refresh token issues a new refresh token and invalidates the old one (refresh token rotation).
- A user can view and revoke applications they have authorized from their security settings, listing the application name, granted scopes, and authorization date. Revocation immediately invalidates all tokens issued under that authorization.
- An admin can view all registered OAuth2 clients, their grant types, active token counts, and can suspend a client (preventing new token issuance and immediately revoking all existing tokens for that client) without deleting its registration.
- OAuth2 scopes map directly to the permission naming convention from section 4.10 Feature 1 (e.g. scope `contacts:read` corresponds to permission `contacts.view`), maintained as a single source of truth to avoid scope/permission drift.

## Feature 4: Webhook system

**Story:** As an admin, I can configure outbound webhooks for CRM events so that external systems are notified in real time when relevant data changes.

Acceptance criteria:
- An admin can register a webhook with a target URL, a list of subscribed event types (e.g. `contact.created`, `deal.stage_changed`, `ticket.resolved`, `contract.signed`, `invoice.paid`), and an optional description.
- Each registered webhook is issued a signing secret at creation, displayed once. Outbound webhook payloads include an `X-Webhook-Signature` header computed as an HMAC-SHA256 of the payload body using the signing secret, allowing receivers to verify authenticity.
- Webhook payloads follow a consistent envelope: `{ "event": "...", "timestamp": "...", "data": {...} }`, where `data` matches the structure of the corresponding API resource for that entity.
- Webhook delivery is fully asynchronous via the `integrations` Horizon queue. The triggering CRM action completes and returns to the user before webhook delivery is attempted — webhook delivery never blocks the user-facing request.
- Failed deliveries (non-2xx response or timeout after 10 seconds) are retried with exponential backoff: 1 minute, 5 minutes, 30 minutes, 2 hours, then marked as failed after the 4th retry. The retry schedule is visible per delivery attempt.
- A delivery log is maintained per webhook showing each attempt's timestamp, event type, response status code, response time, and response body (truncated to 1KB for storage). The log is accessible from the webhook detail view in integration settings.
- A webhook can be paused by an admin without deleting its configuration. Paused webhooks do not receive new event deliveries but their delivery log history is retained. Resuming does not retroactively deliver events missed while paused.
- A failed delivery can be manually retried by an admin from the delivery log, regardless of whether the automatic retry schedule has been exhausted.
- A webhook with more than 50 consecutive failed deliveries is automatically paused, and the admin who registered it receives an in-app and email notification explaining the auto-pause and the failure reason from the most recent attempt.

## Feature 5: Inbound webhook receivers

**Story:** As an admin, I can receive and process inbound webhooks from third-party providers so that events occurring outside the CRM (payments, signatures, messages) are reflected automatically.

Acceptance criteria:
- Dedicated inbound webhook endpoints exist per provider: `/webhooks/stripe`, `/webhooks/twilio`, `/webhooks/docusign`, `/webhooks/mailgun` (or configured email provider), `/webhooks/social/{provider}`, and others as integrations are added — each as a distinct route, not a generic catch-all.
- Each inbound webhook endpoint verifies the provider's signature using that provider's documented signature scheme (e.g. Stripe's `Stripe-Signature` header, Twilio's `X-Twilio-Signature`, DocuSign's HMAC) before any processing occurs. Requests failing signature verification are rejected with 401 and logged as a security event.
- Inbound webhook endpoints respond within 5 seconds (per most providers' requirements) by immediately queuing the payload for processing on the `integrations` Horizon queue and returning a 200 acknowledgement, rather than processing synchronously.
- Inbound webhook endpoints are exempt from CSRF protection (consistent with section 4.10 Feature 10) but are subject to a dedicated, more permissive rate limit appropriate to each provider's expected volume, configurable per endpoint.
- Duplicate inbound webhook deliveries (identified by the provider's event ID, where provided) are detected and ignored idempotently — the payload is recorded as received but no duplicate processing occurs, preventing double-charging, duplicate interaction records, etc.
- A raw inbound webhook log retains the full payload (with sensitive fields masked per section 4.10 Feature 5 classification) for 30 days for debugging, accessible to admins from the integrations settings panel.
- Processing failures for inbound webhooks (e.g. a Twilio call event referencing a contact that no longer exists) are logged with the failure reason and surfaced in an inbound webhook errors panel, without causing the endpoint itself to return a non-200 status (which would trigger the provider's own retry storms).
- New inbound webhook endpoints follow a documented internal pattern (verify signature → queue job → return 200 → process in job) so that adding a new provider integration does not require re-deriving this pattern each time.

## Feature 6: Integration marketplace / pre-built connectors

**Story:** As an admin, I can browse and enable pre-built connectors for common third-party services so that integrations are configured through a guided UI rather than custom development.

Acceptance criteria:
- A connector catalogue lists available pre-built integrations: Mailchimp, Slack, QuickBooks, Xero, Sage, Salesforce (data migration), Google Workspace, Azure AD, DocuSign, Twilio, Africa's Talking, and Stripe — matching the integration points referenced throughout sections 4.4 through 4.10.
- Each catalogue entry shows: connector name, logo, short description, category (communications, finance, productivity, identity, e-signature, payments), connection status (not connected, connected, error), and a primary action button (Connect, Configure, or Reconnect depending on status).
- Connecting a connector launches a guided setup flow specific to that connector's authentication method — OAuth2 redirect for Google/Azure/Xero/QuickBooks/Slack, API key entry for Twilio/Africa's Talking/Mailchimp, and a combination for DocuSign.
- A connected connector's configuration page shows its current settings (sync direction, scopes granted, webhook subscriptions active), a connection health indicator (last successful sync/call timestamp), and a disconnect action.
- Disconnecting a connector is a privileged action (per section 4.10 Feature 4), revokes stored credentials, stops associated scheduled jobs, and is logged in the audit trail. Historical data created by the connector while active is retained.
- The catalogue is searchable and filterable by category. Connectors not yet connected display a "what this enables" summary linking to the relevant feature sections (e.g. Twilio links to the SMS and call logging features in section 4.3).
- Each connector's error state (shown when a previously working connection starts failing, e.g. expired OAuth token) links directly to the reconnection flow and shows the specific error returned by the provider on the last failed attempt.
- The connector catalogue and individual connector configuration pages require the `integrations.manage` permission, assigned to admin role only — consistent with section 4.9 Feature 9 and section 4.4's social media connections.

## Feature 7: API rate limiting per client

**Story:** As an admin, I can configure rate limits per API client so that high-volume integrations do not degrade service for others.

Acceptance criteria:
- This feature extends section 4.10 Feature 9's rate limiting infrastructure with per-OAuth2-client and per-API-key overrides, rather than introducing a separate rate limiting system.
- Each registered OAuth2 client (Feature 3) and each integration API key (Feature 2) can have a custom rate limit configured by an admin, overriding the default `60 requests per minute` general API limit.
- Rate limit overrides are stored alongside the client/key record and read by the same rate limiter middleware via the `rate_limits` configuration table referenced in section 4.10 Feature 9, keyed by client/key identifier rather than route group where an override exists.
- The integration catalogue (Feature 6) and OAuth2 client management screens (Feature 3) display the current effective rate limit for each connected client/integration, with an edit action for admins.
- A client approaching its rate limit (80% of limit consumed within the current window) receives a `X-RateLimit-Warning` response header, in addition to the standard rate limit headers from section 4.10 Feature 9.
- Rate limit override changes are written to the audit trail with the old and new limits and the acting admin.

## Feature 8: OpenAPI documentation

**Story:** As a third-party developer, I can access auto-generated, always-current API documentation so that I can integrate with the CRM without relying on outdated written docs.

Acceptance criteria:
- API documentation is generated using Scribe from route definitions, Form Request validation rules, and API Resource structures, producing an OpenAPI 3.0 specification.
- Documentation is regenerated automatically as part of the CI/CD pipeline on merge to the main branch, ensuring documentation never diverges from the deployed API.
- The developer portal (a section of the integrations settings area, or a standalone `/docs` route) presents the OpenAPI spec via an interactive UI (e.g. Scalar or Swagger UI) allowing developers to browse endpoints, view example requests/responses, and execute test requests against a sandbox environment using their own token.
- Each endpoint's documentation includes: required permissions/scopes, rate limit tier, request/response schemas with field-level descriptions, and example payloads including error responses.
- A "Try it out" feature in the developer portal allows an authenticated developer to execute real API calls using their own personal access token, with a clear visual indication that requests made here affect real data (no separate sandbox database is provisioned for this version).
- The OpenAPI specification file itself is downloadable from the developer portal for import into tools like Postman or Insomnia.
- Breaking changes between API versions (per Feature 1's deprecation policy) are highlighted in a changelog section of the developer portal, generated from version-tagged annotations in the route/controller definitions.
- The developer portal is publicly accessible (no CRM login required) for browsing documentation, but the "Try it out" feature requires authentication with a valid personal access token.

## Feature 9: API service registry

**Story:** As an admin, I can view a registry of all registered integrations and their status so that I have a single place to audit what external systems are connected to the CRM.

Acceptance criteria:
- The service registry is a consolidated view distinct from (but linking to) the connector catalogue (Feature 6) — it lists every integration record regardless of whether it originated from a catalogue connector, a custom webhook, an OAuth2 client, or an API key.
- Each registry entry shows: integration name, type (catalogue connector, custom webhook, OAuth2 client, API key, inbound webhook receiver), status (active, paused, error, revoked), last activity timestamp, and a link to its detailed configuration/log view.
- The registry supports filtering by type and status, and a search by integration name.
- An "all activity" view aggregates recent events across all registry entries — outbound webhook deliveries, inbound webhook receipts, OAuth2 token issuance, and sync job runs — into a single chronological feed for quick health-checking.
- The registry highlights integrations with no activity in the last 30 days as potentially stale, with a suggestion to review whether they are still required.
- Exporting the full registry as CSV is available to admins for compliance and security review purposes (e.g. as input to the GDPR/PDPA data processing register required under section 4.10 Feature 6).
- The registry requires the `integrations.manage` permission, assigned to admin role only, consistent with all other integration management features.

## Feature 10: Event-driven integration architecture

**Story:** As a developer, I need all integrations to fire through queued events rather than blocking requests so that the CRM remains responsive regardless of third-party system performance.

Acceptance criteria:
- Every integration touchpoint referenced across sections 4.4–4.10 (campaign dispatch, webhook delivery, accounting sync, e-signature requests, SMS/email sending, social posting) is implemented as a Laravel event with one or more queued listeners — no integration call is made synchronously within an HTTP request/response cycle, except the OAuth2 authorization flows (Features 3 and 6) which are inherently synchronous redirects.
- All integration-related queued jobs run on the dedicated `integrations` Horizon queue, isolated from `default`, `campaigns`, `analytics`, and `imports` queues, so that a slow or failing third-party system does not starve other job types of worker capacity.
- Each integration job implements Laravel's `ShouldQueue` with a configured `tries` and `backoff` matching the retry behaviour specified for that integration elsewhere in the specs (e.g. webhook delivery's 1m/5m/30m/2h schedule from Feature 4, accounting sync's exponential backoff from section 4.9 Feature 9).
- A failed integration job that exhausts all retries is moved to the `failed_jobs` table and surfaced in the relevant integration's error panel (webhook delivery log, sync log, etc.) — not just in Horizon's generic failed jobs view, which remains for admin-level operational monitoring only.
- The `HttpClientService` circuit breaker (referenced in section 4.9 Feature 9) is the mandatory entry point for all outbound third-party HTTP calls across every integration — no integration service calls an external HTTP API directly without going through this service, ensuring consistent timeout, retry, logging, and circuit-breaking behaviour.
- Horizon's dashboard is accessible to admins for monitoring queue depth, job throughput, and failed job counts per queue, with the `integrations` queue's metrics specifically surfaced on the system health panel of the admin dashboard (per section 4.7 Feature 1's admin dashboard).

## Your questions

**Is this a marketplace of connectors on the UI?**

Yes — Feature 6 is exactly that. It's a catalogue/grid UI (logo, name, description, category, status, connect/configure button) similar to what you'd see in Slack's or HubSpot's app directories. It's the primary entry point for non-technical admins to enable Mailchimp, Slack, QuickBooks, Twilio, etc. without touching API keys directly where OAuth is available.

**Should it be in the settings menu?**

Yes, and it should be the umbrella for almost everything else in 4.11 from a navigation perspective. I'd structure the settings navigation like this:

**Settings → Integrations** (top-level section), containing:
- **Connector marketplace** (Feature 6) — the visual catalogue, the default landing tab
- **API access** — Sanctum tokens and OAuth2 client management (Features 2 and 3), split into "My tokens" (personal, any user) and "OAuth2 applications" (admin-only)
- **Webhooks** — outbound webhook registration and delivery logs (Feature 4)
- **Service registry** — the consolidated audit view (Feature 9), probably as an "Overview" or "All integrations" tab since it spans everything else
- **Inbound webhook logs** — Feature 5's raw payload log, likely nested under the service registry or its own tab since it's diagnostic rather than configuration
- **Developer portal** link — Feature 8's docs, which could live partly outside settings (public `/docs` route) but with a settings shortcut for admins

The rate limiting overrides (Feature 7) don't need their own nav item — they're best surfaced inline on each connector/client's configuration page, as the specs describe, rather than as a separate settings screen.

One structural note: Features 2's "My tokens" panel is the one piece every user sees, not just admins. Everything else under Integrations should be gated behind `integrations.manage` (admin-only), so you'll likely want "API access → My tokens" reachable from general user settings, while the rest of the Integrations section is hidden from the settings nav entirely for non-admin roles.
